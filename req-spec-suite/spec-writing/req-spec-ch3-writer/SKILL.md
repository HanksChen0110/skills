# 需规生成器（第 3 章）

用于生成、改写或修订中文需求规格说明书第 `3` 章，尤其是 `3.2 CSCI能力需求` 及其配套表格。该 skill 的目标不是自由发挥，而是在既有模板和前文章节已确认的前提下，把第 3 章写成“命名统一、结构完整、表格合规、描述可验收”的交付稿。

## 先读文件

- 先读用户提供的模板 `.docx`。
- 再读目标需求规格说明书 `.docx`。
- 必须回看前文 `1` 章至 `3.1.3`，确认系统名称、子系统名称、角色、边界、对象、术语已经定义。
- 再读本 skill 目录下的 `references/需规写作宪法.md`。
- 必要时联动 `req-spec-reviewer` 做审稿式核查。
- 用户在乎 Word 版式时，联动 `doc` 或 `docx` 负责格式落地。

## 写作边界

- 本 skill 负责 `3.2` 为主，也可修补第 3 章内与 `3.2` 强关联的小节。
- 不要在第 3 章凭空发明前文没有出现过的新子系统、新角色、新命名。
- 如果前文没有定义而第 3 章又必须使用，先回补前文，再回写第 3 章。

## Chapter 3 Core Rule

- 第 3 章必须从前文总体结构自然展开。
- `3.2.1`、`3.2.2`、`3.2.3` 等子系统章节的名称、标识、边界，应与前文完全同名。
- 后文出现的功能分层、用例分层、接口对象、规则对象，都要能在前文找到来源。
- 如果前文已建立“业务环节总层 -> 系统模块层”的映射，则 `3.2` 的每个模块总述要延续这个映射，说明该模块主要支撑哪些业务节点，但不要借机扩张建设范围。

## 功能分解与用例分解

- `功能分解`、`用例分解` 这类内容如果是正式分节，必须写成带编号的小节。
- 四级标题如 `3.2.3.1 功能分解`、`3.2.3.2 用例分解` 一律使用 `黑体`，不要使用 `微软雅黑`。
- 不要留下无编号的悬浮标签式段落标题。

## 表格规则

- 表头文字必须与模板完全一致。
- 不要改表头列名，不要擅自新增列、删列、换列名。
- 表题必须放在表格外部，不能写进表格的第一行、第一格或表头单元格。
- 如果遇到“表题已经跑进表内”的文档，优先把表题抽到表格外部，而不是改写表头内容。
- 如果表格本身第一行已经是正常表头，不要再把 `表 X ...` 写入表中。
- 不允许把前后两张表连成一张表。
- 不允许出现孤行空记录，例如只有一个编号或标识如 `AUDIT-A-01-01~03` 单独占一行，其他列为空。这类情况要判断是漏写、错位还是误合并，并修正成完整记录。

## 功能描述写法

- 功能描述必须自然覆盖以下信息，但不要把英文术语直接写进正文：
  - 在什么业务场景下触发
  - 在什么环境或节点发生
  - 面向什么角色或对象
  - 系统具体执行什么动作
  - 解决什么业务问题或满足什么目的
  - 最终产生什么结果或影响
- 换句话说，描述要具备 5W + Effect 的信息量，但正文不要出现 `when / where / who / what / why / effect` 这些英文标签。
- 不要写成空泛套话，例如“用于实现 xx 管理”“负责 xx 相关业务处理”“支撑 xx 业务开展”。

## 内容组织规则

- `3.2` 应保持从“子系统总述”到“功能分解”再到“用例分解”的层次。
- 如果模板允许表格表达，优先用表格承载结构化信息。
- 如果正文出现大段密集说明且不利于阅读，可拆成短段，但不要破坏模板规定的章节结构。
- 用户若明确要求“尽量不要分点描述”，则保持成自然段；若用户要求减少大段文字，可适度拆解为短条目。始终服从当前任务要求。
- 模块总述不应只是“该模块负责什么”。更好的写法是：在什么业务节点或运行阶段触发、面向谁、处理什么对象、产出什么结果、如何与其他模块闭环。

## 常见修订场景

- 修正 `3.2.x.1 功能分解` 等四级标题字体错误。
- 补齐表题与表格分离。
- 修正表中错位、空行、孤立标识、串行串列问题。
- 按前文已定义的总分结构，重写 `3.2` 表格中的功能描述。
- 把功能描述改成有业务场景、有对象、有动作、有结果的正式表述。
- 当子系统数量从三模块扩展为四模块或更多时，必须同步检查 `3.1` 总体框架、`3.1` 动态关系、`3.1.3` 能力配置、`3.2` 子系统章节数，以及图题表题编号是否全部跟上。

## Word 排版协同规则

- 四级标题样式与字体由 `doc` 或 `docx` 在 Word 中落地。
- 目录更新后，只允许真实标题进入目录。
- 功能表或用例表修订后，要复查是否误把普通段落设置成标题级别。

## 自检清单

- 第 3 章是否只使用前文已定义的命名。
- `功能分解`、`用例分解` 是否为正式编号小节。
- 四级标题是否为 `黑体`。
- 表题是否全部在表格外部。
- 是否存在孤行编号、空行、错位、串表。
- 功能描述是否具备完整业务语义，而不是空话套话。
- 目录是否未被正文污染。
