# 需规后半段模板规则

## 适用范围

本规则用于生成或续写需求规格说明书 `3.3` 至 `7` 的剩余章节。

## 一、结构继承规则

当存在原始需规、`docx` 或模板 `pdf` 时，默认按原文格式续写，至少继承以下要素：

- 章节编号和层级深度
- 标题命名风格
- 图题、表题和表号风格
- 表号是否通过题注或域自动生成
- 表格列头和列顺序
- 正文描述口吻
- 术语、角色、模块、对象、接口和数据名称

禁止事项：
- 擅自改变 `3.3 ~ 7` 的章节顺序
- 擅自把表格需求改写成自由散文
- 擅自把正式需规语言改成方案汇报语言
- 将原文依赖域自动生成的表号改为手工输入，或在同一文档中混用两种表号方式

## 二、接口章节规则

### 3.3 外部接口需求

重点回答：
- 系统与哪些外部对象交互
- 交互边界在哪里
- 交互输入输出是什么
- 采用什么通信、协议、频率或约束

默认不展开：
- 全量字段字典
- 全量报文示例
- 代码级接口实现

### 3.4 内部接口需求

重点回答：
- 模块之间如何调用和衔接
- 接口名称、标识、触发方向和调用条件
- 接口图或接口关系说明

若接口条目涉及分期范围、智能体对接、硬件接入或其他扩展能力：
- 备注栏必须明确当期交付状态

## 三、内部数据章节规则

### 3.5 内部数据需求

重点写：
- 数据对象之间的关系
- 数据来源与去向
- 主键对象、状态对象、结果对象之间的关联
- 文本输入统一规则、口径统一规则、命名规则

不要直接退化成数据库设计说明书。

## 四、环境与质量章节规则

### 3.6 环境需求

至少覆盖：
- 硬件环境
- 软件环境
- 网络环境
- 其他环境

写法要求：
- 明确最低条件、依赖条件、部署边界和适配条件
- 不写无依据的高配建议

### 3.7 其他质量特性需求

常见子项包括：
- 可靠性需求
- 维修性需求
- 安全性需求
- 测试性需求
- 保障性需求
- 适应性需求
- 人机环境系统工程设计

写法要求：
- 每项都尽量对应可验证标准
- 优先使用“应能够”“应支持”“应保证”“应满足”这类可验收表达
- 不使用“首先、其次、最后”等口语化过渡词
- 不使用“非常、十分、极其”等程度副词

## 五、约束与优先级章节规则

### 3.8 计算机资源需求

说明系统对 CPU、内存、存储、网络、并发或其他关键资源的需求边界。

### 3.9 设计和实现约束

说明：
- 技术栈约束
- 标准规范约束
- 合规约束
- 数据交换约束
- 部署或集成约束

### 3.10 需求的优先顺序和关键程度

至少说明：
- 需求优先级顺序
- 关键需求、重要需求和一般需求的划分依据

当条目涉及分期建设、扩展能力或后续版本时，使用以下状态说明：
- 本期建设：正常描述
- 架构预留：标注“本期完成架构预留，待后续版本补充交付”
- 不纳入本期：标注“本期不纳入建设范围，预留接口，后续版本扩展”

## 六、收口章节规则

### 4 合格性规定

必须说明每类需求采用何种验证方法，例如：
- 评审
- 分析
- 检查
- 演示
- 测试

### 5 需求可追溯性

至少建立以下一种追溯关系：
- 需求来源 -> 需求条目
- 需求条目 -> 设计章节
- 需求条目 -> 测试项
- 需求条目 -> 交付物

### 6 注释

常用于收纳：
- 术语及缩略语
- 业务规则或公式
- 界面需求
- 导出文档/报表模板格式
- 异常处理

### 7 交付物清单

至少说明：
- 交付物名称
- 交付阶段
- 交付内容说明

若原文已有表格结构，严格保留。
若原文表号依赖 Word 域或题注自动生成，续写时必须继续使用该机制，不手工补写表号文本。

## 七、表格与表述规则

在无更强原文约束时：
- 表题统一采用 `表X.X.X 表格名称`
- 表题单独成行并居中
- 备注栏用于说明扩展能力、交付状态和范围约定，不写空泛备注

正文写法要求：
- 保持正式、客观、工程化表述
- 不出现 `what`、`how`、`why`、`who`、`when` 等分析框架字样
- 不在正文中堆砌背景分析或行业综述
