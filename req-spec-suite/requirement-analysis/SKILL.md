# 需求分析 (Requirement Analysis)

本Skill提供一套标准化的需求分析与评估框架。当被要求“分析一下这个需求”、“判断这个功能值不值得做”、“对以下需求进行优先级排序”时，必须严格按照本指南进行深度思考和输出。

## 1. 需求分析思考方法论

简单思考总结为：为什么要做，谁要做，在什么情景做，需要得到什么结果，如何跟踪数据以确定目标的达成。

进行需求拆解时，必须涵盖以下6个维度：

- **背景 (Background)**
  - 用户目前遇到了什么问题？用户想要什么？为什么需要？
  - 是否来源于企业战略规划（如企业数字化转型、GPT接入、自研细分模型等）？大背景下会产生哪些应用需求？
- **用户 (User)**
  - 提出需求的用户是谁？需求上线后真正使用的人是谁？
  - 必须分清楚“提出人”和“使用人”，站在他们的实际角度分析需求。
- **情景 (Scenario)**
  - 用户在什么场景下适用？
  - 这个场景是否真实存在？场景的适用频次是否属于高频？
- **目标 (Goal)**
  - 用户期望达到的最终效果是什么？什么是挖掘出的真实用户诉求？
- **数据 (Data)**
  - 用户操作会产生什么数据？这些数据有什么业务价值？
  - 如何通过数据采集跟踪目标达成？（注意：C端产品重用户行为数据，B端产品重业务数据）。

## 2. Kano模型 (Kano Model) 需求分类

在完成需求拆解后，使用Kano模型对该需求进行属性界定，以判断其对用户满意度的影响及优先级：

- **必备型需求 (Must-be, M)**
  - 表现：需求满足时，用户不会感到满意；需求不满足时，用户会很不满意。
  - 策略：痛点，必须做，做深做透。
- **期望型需求 (One-dimensional, O)**
  - 表现：需求满足时，用户会感到很满意；需求不满足时，用户会很不满意。
  - 策略：线性需求，是竞品拉开差距的核心，按资源投入。
- **魅力型需求 (Attractive, A)**
  - 表现：该需求超过用户对产品本来的期望，使得用户的满意度急剧上升；即使表现的不完善，用户的满意度也不受影响。
  - 策略：爽点/Aha Moment，寻求低成本高杠杆的创新切入。
- **无差异型需求 (Indifferent, I)**
  - 表现：需求被满足或未被满足，都不会对用户的满意度造成影响。
  - 策略：多为伪需求或极客小众需求，建议砍掉或延后。
- **反向型需求 (Reverse, R)**
  - 表现：该需求与用户的满意度呈反向相关，满足该要求，反而会使用户的满意度下降。
  - 策略：画蛇添足，绝对不做。

## 3. 分析输出标准模板

在为用户进行需求分析时，请直接应用以下结构化输出模板：

```markdown
### 💡 需求价值判定结论
*(一句话总结：这是不是一个真需求，核心解决什么干系人的痛点，Kano属性是什么，建议的优先级/Action是什么。)*

### 🛠️ 1. 结构化需求拆解
- **背景溯源**: *(分析动机与背景...)*
- **干系人分析**: *(提出人是谁，实际使用者是谁...)*
- **场景频率**: *(什么情况下用，高频还是低频...)*
- **核心目标**: *(预期达成的真实效果...)*
- **数据闭环**: *(产生什么业务/行为数据，如何衡量成功...)*

### 📊 2. Kano模型评估
- **需求属性界定**: `[M必备型 / O期望型 / A魅力型 / I无差异型 / R反向型]`
- **判定理由**: *(为什么属于该分类...)*
```
