# 需规写作宪法

## 定位

这份文档用于约束中文需求规格说明书，尤其是基于 Word 模板交付的正式需规，解决两个最常见的问题：

- 文字内容看似完整，但缺少“业务总层 -> 系统切入点 -> 系统模块层”的总分结构。
- 版式视觉上像文档，但 Word 语义上不是正式交付稿，章节、题注、目录、正文样式混在一起。

目标不是把文档写得更花，而是把文档写得更像“可交付、可维护、可扩展、可继续修订”的正式规格说明书。

## 第一原则：先立总纲，再写系统

前文不能一上来就写“系统有哪些模块”“系统做什么功能”。

标准顺序应优先采用：

1. 业务全貌：先讲项目所在的业务主链路、行业背景、总体业务盘子。
2. 切入节点：明确系统赋能的是哪几个业务环节、哪类节点、为什么从这里切入。
3. 样板场景：用一个典型场景说明痛点、方案和价值闭环。
4. 底座与差异化：最后再讲平台底座、数据来源、接口条件和系统相对优势。

如果系统只是赋能某些审核/确认节点，必须明确写出：

- 系统不替代业务系统本体
- 系统不替代业务算法本体
- 系统重点赋能的是关键审核节点、确认节点、留痕节点、复盘节点

## 第二原则：业务层和系统层必须分开表达

正式需规最容易犯的错，是把“业务层”与“系统层”写混。

必须显式区分：

- 业务层：系统赋能哪些业务环节，当前重点落在哪些环节，延伸覆盖哪些环节。
- 系统层：系统内部由哪些模块承接这些业务落点。

推荐写法：

- 先写“业务总层”
- 再写“重点落点 + 延伸环节”
- 再写“系统模块层”

不要把“系统有四个模块”误写成“系统覆盖整个业务链”。

## 第三原则：前文必须给后文兜底

`1 ~ 3.1.3` 的职责不是凑字数，而是给 `3.2` 和后续章节立前提。

前文必须提前定义清楚：

- 系统名称
- 子系统名称
- 角色名称
- 边界对象
- 外部接口对象
- 数据对象
- 核心术语
- 模块间总体关系

后文不允许突然冒出前文没有铺垫的新模块、新角色、新边界。

## 第四原则：3.2 必须从 3.1 自然展开

`3.2 CSCI能力需求` 不是重新开一篇新文档，而是对 `3.1` 已经确定的总体结构做展开。

每个模块章节至少要有：

- 模块总述
- 功能分解
- 用例分解

模块总述不能只写“负责 xx 管理”，而要自然覆盖：

- 在什么业务节点或运行阶段触发
- 面向谁
- 处理什么对象
- 做什么动作
- 产出什么结果
- 与哪些模块形成闭环

如果模块数量变化，例如从三模块扩成四模块，必须同步更新：

- `3.1` 总体功能框架
- `3.1` 动态关系
- `3.1.3` 能力配置
- `3.2` 子系统数量与顺序
- 图题、表题、角色关系、编号顺序

## 第五原则：正式需规优先采用高密度自然段

正式需规不是会议纪要，也不是草稿大纲。

正文应优先使用：

- 完整自然段
- 更高信息密度
- 更稳定的工程语气

应避免：

- 大量一两行短句切碎正文
- 标题、题注、章节号、说明句都混成 `Normal`
- 读起来像“正在整理的草稿”

判断标准很简单：

- 优化后的正式稿，通常“段落更少，但每段更密”
- 草稿化文档，通常“段落很多，但信息很散”

## 第六原则：Word 语义结构比视觉相似更重要

“看起来像模板”不等于“就是正式 Word 交付稿”。

正式需规必须满足：

- 真正的章节标题用真正的 `Heading`
- 真正的图题、表题用真正的题注段落
- 真正的正文使用正文样式，如 `Normal Indent`
- 目录由真实标题自动驱动
- 标题和题注不得继承 Office 默认主题样式中的蓝色、紫色、斜体或装饰性强调

不允许出现以下伪结构：

- `3.2 CSCI能力需求` 写成普通正文
- `4 合格性规定` 写成普通正文
- `表 7 xxx 一览表` 写成普通正文
- 图题表题写进表格内部
- 正文误带大纲级别进入目录

一句话总结：

- 标题必须是真的标题
- 题注必须是真的题注
- 正文必须是真的正文
- 标题和题注还必须是“标准稿风格”的标题和题注，而不是 Office 默认主题标题

## 第六点一原则：必须显式覆盖默认主题样式

使用 `python-docx`、`docx-js` 或其他方式生成 Word 时，不能依赖内建默认 `Heading 1~4` 与 `Caption` 样式，因为它们常常带有：

- 蓝色主题色
- 自动粗体
- 四级标题斜体
- 与正式需规不一致的字号和间距

因此必须显式覆盖：

- `Heading 1 ~ Heading 4`
- `Caption`
- `Normal Indent`

最低约束如下：

- 各级标题字体、字号、颜色、斜体、对齐方式要与标准稿一致
- 标题默认颜色应为黑色
- 标题不得使用斜体
- 表题、图题不得使用蓝色或其他主题色
- 题注不应使用斜体
- 正文样式要统一到正式正文样式，不能只靠段落局部格式补丁

进一步落地时，应至少逐项比对：

- 一级至四级标题的字体、字号、颜色、粗斜体状态
- 表题、图题的字体、字号、颜色、粗斜体状态
- 正文段落的字体、字号、首行缩进、对齐方式
- 表头行的对齐方式
- `序号` 列的对齐方式
- `名称/标识` 列的对齐方式
- `描述` 类列的对齐方式

如果标准稿中标题是黑体、小四、黑色、非斜体，那么生成稿也必须逐项一致，不能只做到“看起来差不多”。

## 第七原则：目录只能反映真实结构

目录不是装饰，而是结构验收工具。

目录必须满足：

- 只包含真实标题
- 不包含正文
- 不包含题注
- 不包含列表项

常见问题与判断：

- 如果正文里有 `4 合格性规定`，目录里却没有，说明这是伪标题。
- 如果目录里出现整句正文，说明正文被误赋予了标题样式或大纲级别。

每次结构调整后都要重新核查目录。

## 第八原则：图题表题必须在表外、图外

图题和表题是交付语义的一部分，不能混进正文，也不能混进表格内部。

必须遵守：

- 图题单独成段
- 表题单独成段
- 表头只保留表头，不承担表题功能
- 图题、表题颜色统一为黑色，不使用主题蓝色
- 图题、表题不使用斜体

禁止：

- 把 `表 X ...` 写进表格第一行
- 把 `图 X ...` 混成正文说明句
- 前后两张表误连成一张

## 第八点一原则：表格排版要按“列角色”微调，而不是整表一把梭

正式需规中的表格，不是所有列都统一居中，也不是所有列都平均分宽。

要先判断每一列承担的角色，再决定列宽和对齐方式。

常见列角色与推荐规则如下：

- `序号`、`章节号`、`优先级`、`版本号`、`日期` 这类短字段列：列宽收窄，内容居中。
- `角色`、`功能名称/标识`、`文档标识` 这类短标识列：如果内容短且规整，优先居中；如果内容开始变长、阅读上出现拥挤，则允许左对齐。
- `功能描述`、`内容描述`、`功能说明`、`用例目标`、`解释`、`传统处理方式`、`变化说明` 这类叙述列：优先给最大宽度，正文左对齐。
- `备注` 列：如果主要是“无 / 主链路 / 可选”这类短值，可居中；如果备注已经承载解释性文字，应改为左对齐。

列宽分配也应遵守同样逻辑：

- 不要机械平均分列宽。
- 叙述列应明显宽于标识列。
- 当一张表中某一列承担主要阅读压力时，应主动压缩其他短字段列，把宽度让给该列。
- 只有在表格本身就是对称结构、且各列信息密度接近时，才可以采用近似均分列宽。

从已优化标准稿中可以沉淀出以下经验：

- 表格整体通常保持居中放置。
- 表格一般关闭自动自适应，尽量使用明确列宽。
- 表头行统一居中。
- 数据行不要默认全列居中，而应按列角色分配左右对齐。
- 多行文本单元格不要强制做垂直居中，否则视觉上会发飘；正文型表格更适合保持自然顶部对齐。

一句话约束：

- 先判定列角色，再决定列宽和对齐；不要用“整表统一居中 + 平均分列”替代排版判断。

## 第九原则：尾章不是占位，而是交付闭环

`3.3 ~ 7` 不能写成“有标题就算完成”。

尤其要保证：

- `4 合格性规定` 有可执行、可检查内容
- `5 需求可追溯性` 完整
- `6 注释` 不留大面积无效占位
- `7 交付物清单` 明确

同时，尾章也必须保持真实 Word 结构：

- 章节是真的章节
- 表题是真的题注
- 正文是真的正文

## 第十原则：需规必须能承受后续增量修订

一份好的需规，不只是当前看着顺，更要方便下一轮扩展。

因此在写法上要主动保证：

- 命名统一
- 章节稳定
- 总分关系稳定
- 图表编号稳定
- 模块增删时可整体联动

文档不能写成“一处改了，其他地方全断裂”的状态。

## 推荐写作骨架

### 对于 `1.2 项目背景 / 系统概述`

推荐顺序：

1. 行业背景 / 业务背景
2. 需求分析
3. 公司背景 / 平台背景
4. 当前状态
5. 建设必要性

### 对于 `业务范围`

推荐结构：

1. 先写系统赋能哪些业务环节
2. 再写当前重点落点
3. 再写延伸范围
4. 再写系统内部模块承接关系

### 对于 `3.1`

推荐顺序：

1. 要求的状态和方式说明
2. 功能框架及关系描述
3. 状态和方式下能力配置

### 对于 `3.2`

每个模块建议固定结构：

1. 模块总述
2. 图
3. 功能分解
4. 用例分解

## 表格排版补充规则

正式需规中的表格，除内容正确外，还应满足基本排版一致性：

- 表头行文字居中
- `序号` 列居中
- `名称/标识` 类列通常居中或按标准稿要求对齐，不得随意漂移
- 描述类列按可读性左对齐或两端对齐
- 表头字体、字号、粗细与标准稿一致
- 不能让表格保留明显的 Office 默认风格痕迹

## 终审检查清单

交付前至少逐项确认：

- 是否先讲清楚业务全貌，再讲系统切入点
- 是否明确写出系统赋能的是哪些业务环节、哪些节点
- 是否区分了业务层与系统层
- 是否所有后文模块都已在前文出现
- 是否 `3.2` 与 `3.1` 完全对齐
- 是否模块数量变化已同步到角色、图题、表题、动态关系
- 是否章节全是 Word 真标题
- 是否图题表题全是独立题注
- 是否目录只包含真实标题
- 是否正文仍保持正式需规的自然段节奏

## 一句话标准

一份优秀的需规，不只是“内容写全”，而是同时满足这四点：

- 逻辑上有总分
- 架构上能承接
- Word 结构上是真的
- 阅读节奏上像正式交付稿
