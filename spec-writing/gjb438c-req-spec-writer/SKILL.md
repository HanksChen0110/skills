# GJB 438C 需规生成器

模板参考：
`D:/AIwork/pm-workbench/skills/spec-writing/gjb438c-req-spec-writer/references/公司需求规格说明书模板.docx`

用于生成、续写、改写或统一整份中文需求规格说明书，覆盖 `1 ~ 7` 全部章节，并严格以用户给定的 Word 模板为唯一版式基准。

## 先读文件

- 先读用户提供的正式 `docx` 模板或既有成稿
- 再读：
  - `D:/AIwork/pm-workbench/skills/spec-writing/template-first-req-spec-rules.md`
  - `D:/AIwork/pm-workbench/skills/spec-writing/req-spec-hard-constraints.md`

- 必要时再读本 skill 目录下的 `references/`

## 工作顺序

1. 识别任务类型：新写、续写、重构、按模板回填、按配置清单对齐
2. 提取模板骨架：标题层级、表头、图题、表题、编号方式、页眉页脚口径
3. 整理业务流程和模块树，至少确认到三级功能
4. 对照配置清单、整改要求、评审意见，形成“保留、微调、重写”清单
5. 分批生成：
   - 第 1 批：`1 ~ 3.1.3`
   - 第 2 批：`3.2`
   - 第 3 批：`3.3 ~ 7`
6. 每完成一批，必须按 `req-spec-reviewer` 规则自审
7. 交付前必须做模板完整性检查和表格观感检查

## 黄金准则

- 以真实 Word 结构交付，不以“视觉上像模板”交付。章节、题注、目录都必须由正确样式驱动。
- 前文先建立业务总盘子，再写系统切入点，再写样板场景，再写平台底座与差异化，不要开篇就陷入系统自我介绍。
- 若系统只赋能某些业务环节或节点，必须明确“业务总层 -> 当前重点落点 -> 系统模块层”的映射，不得把局部能力写成全业务覆盖。
- 正式需规应采用更高信息密度的完整自然段，避免把正文切成大量草稿式短句。

## 模板硬约束

- 模板优先，不得把正式需规写成“像模板的普通文档”
- 不得修改模板固定表头、固定章节名、固定图题和固定表题口径
- `功能划分一览表` 第四列即使语义上承担功能描述作用，也必须保留模板原名 `备注`
- `3.2` 只能在一级、二级、三级功能树确认后展开
- `4 合格性规定`、`5 需求可追溯性`、`7 交付物清单` 不得缺失
- 如果模板将追溯拆为 `正向追溯`、`反向追溯`，则两节必须同时落位
- 不得出现“目录是新的，正文还是旧的”或“正文改了，表格没改”的混合状态
- 不得出现“章节号写在正文里，但 Word 并未把它当标题”的伪章节
- 不得出现“图/表题写在正文里，但 Word 并未把它当题注”的伪题注

## 功能描述强约束

以下位置都按正式需规口径书写，不得写成低信息量套话：

- 一级模块概述
- 二级模块说明
- `功能划分一览表` 的 `备注`
- `功能分解` 表中的 `功能描述`
- `功能说明` 表中的 `功能说明`

每条描述至少要自然覆盖以下要素：

- 服务对象是谁
- 在什么业务环节触发
- 输入来源是什么
- 系统做什么处理
- 输出什么结果
- 解决什么痛点或达到什么目的
- 异常或人工确认怎么处理

直接判为不合格的写法：

- “负责 xx 相关业务表单、过程留痕和结果管理”
- “用于实现 xx 管理”
- “支撑 xx 业务处理”
- 只替换名词、不提供行业语义的模板句

## 用例编写约束

- 用例表不能只写一句泛化描述
- 默认至少检查并生成 `新增 / 修改 / 删除 / 查询`
- `其他` 用例按业务特性决定，可写可不写；若该功能存在复核、确认、发布、回填、退回、锁定、归档、审批等特有动作，应补充 `其他`
- 用例目标应写出：
  - 录入或触发动作
  - 关键校验或约束
  - 处理结果
  - 异常或阻断条件
- 用例表禁止串列、错位、自动编号残留或标题内容进入角色列

## 基础信息和类库章节约束

- `3.2.1.6.2` 往下的基础信息类库不得只列标题而无内容
- 若类库型功能不适合写业务用例，必须补充参数项、字段项、规则项或枚举项表格
- 参数表至少要体现：
  - 参数名称或字段名称
  - 标识
  - 参数说明
  - 是否必填或备注
- 必须让审阅者看得出“参数是什么、范围是什么、如何被系统使用”

## 表格格式约束

- 表格边框统一为：外框加粗、首行下边框加粗、内部细线
- 表格首列编号统一写显式 `1/2/3/...`，不得混入自动编号残留
- 列宽要服务于阅读，不得平均分配
- 原则上：
  - `序号/优先级/备注` 列窄
  - `功能描述/功能说明/用例目标/追溯备注` 列宽
- 正向追溯和反向追溯表也要单独调列宽，不可沿用默认平均宽度

## 术语和命名约束

- 若项目口径已确认，则全文统一使用该口径，不得旧名词与新名词混用
- 当前已确认的典型口径包括：
  - `探矿管理系统（本体）` -> `动态本体子系统`
  - `探矿管理系统（智能体）` -> `智能体子系统`
- 一旦口径变更，目录、正文、表格、追溯、图题、表题必须同步更新

## 3.2 专项规则

- `3.2` 严格按确认后的一级、二级、三级功能树展开
- 每个模块至少检查：
  - 模块说明
  - 功能分解
  - 用例分解
  - 对应表格
- 不允许一级功能和二级功能同时重复展开同一批功能点
- 功能树、正文说明、功能表、用例表四者必须一致
- 智能体模块也必须具备完整的功能说明和用例说明，不能只列名称
- 如果前文已经声明业务总层，例如若干业务环节或主流程阶段，则 `3.2` 中每个模块总述应回扣其支撑的业务节点，但不得越权扩张建设范围

## 尾章规则

`3.3 ~ 7` 必须保持工程文档口径，不写成业务汇报材料。重点检查：

- 接口边界是否清晰
- 数据规则是否落位
- 环境与质量要求是否可验证
- 合格性、追溯性、交付物是否完整

## 交付前检查

- 模板结构是否保持
- 表头是否被改名
- `备注/功能描述/功能说明` 是否写实
- 用例是否存在错位、串列、列宽失衡
- 基础信息类库是否补了参数项或等价约束内容
- `5 需求可追溯性` 是否完整
- `6 注释` 是否仍有大面积无效占位
- 是否给出最终可用文档的绝对路径
