# 产品经理工作助手

## 概述

本技能提供产品经理工作的完整方法论，基于《用户体验要素》和《启示录：打造用户喜爱的产品》的核心方法论提炼，涵盖产品机会评估、产品定义、MVP策略、PMF验证等核心知识。

## 一、产品机会评估：十问模型

### 核心原则

1. **聚焦问题**: 先定义问题，再思考解决方案
2. **用户导向**: 明确目标用户是谁，他们有什么痛点
3. **市场验证**: 评估市场规模和竞争环境
4. **能力匹配**: 为什么我们最适合做这件事
5. **时机判断**: 现在做是否是最佳时机

### 机会评估十问

1. 解决什么问题？（用户痛点是否真实存在）
2. 目标用户是谁？（用户细分是否清晰）
3. 市场规模多大？（市场天花板在哪里）
4. 为何我们最适合做？（核心竞争优势是什么）
5. 时机是否合适？（市场是否成熟）
6. 如何验证需求？（用什么方法验证假设）
7. 商业模式是什么？（如何盈利）
8. 技术可行性如何？（技术实现难度）
9. 成功标准是什么？（如何衡量成功）
10. 风险有哪些？（最大风险是什么）

### 评估检查清单

- [ ] 用户痛点是否真实存在（观察行为 > 听用户说）
- [ ] 用户是否愿意为此付费（伪需求三标准：无痛点、低频次、不愿付费）
- [ ] 市场规模是否足够大
- [ ] 团队是否有能力执行
- [ ] 时机是否成熟

## 二、产品定义：五层模型框架

### 核心原则

1. **自下而上**: 每一层决策依赖下层基础
2. **双重平衡**: 平衡用户需求与商业目标
3. **文档驱动**: 用文档确保团队一致性
4. **验证迭代**: 通过用户测试验证假设
5. **量化描述**: 避免主观词，用数据说话

### 2.1 战略层（Strategy Plane）

**核心任务**：
- 定义产品目标：我们要通过这个产品得到什么？
- 定义用户需求：用户要通过这个产品得到什么？

**关键产出**：
- 战略文档：明确平衡用户需求与商业目标的愿景
- 用户画像（Persona）：基于真实用户访谈创建
- 用户场景：描述用户在什么情况下使用产品

**检查清单**：
- [ ] 产品目标是否可量化（如用户留存率、转化率）
- [ ] 用户需求是否基于真实调研（问卷/访谈）
- [ ] 是否创建了用户画像
- [ ] 用户需求与商业目标是否平衡

### 2.2 范围层（Scope Plane）

**核心任务**：
- 功能规格：定义产品功能清单
- 内容需求：确定内容类型和更新频率

**功能规格三原则**：
1. **乐观性**: 引导行为而非禁止（如"未登录时引导登录"而非"禁止发言"）
2. **具体性**: 量化描述（如"点赞数＞1000显示热门标签"）
3. **客观性**: 避免主观词（如"时尚"），改用"符合品牌指南"

**检查清单**：
- [ ] 功能规格是否符合三原则（乐观性、具体性、客观性）
- [ ] 是否明确了MVP功能范围
- [ ] 是否防止了范围蠕变
- [ ] 功能优先级是否清晰

### 2.3 结构层（Structure Plane）

**核心任务**：
- 交互设计：构建概念模型，设计错误处理机制
- 信息架构：组织节点关系，设计导航结构

**信息架构类型**：
- 层级结构：树状分类
- 矩阵结构：多维度筛选
- 自然结构：自由探索
- 线性结构：单一路径

**检查清单**：
- [ ] 概念模型是否符合用户默认习惯
- [ ] 是否设计了错误处理机制（预防-纠正-恢复）
- [ ] 信息架构是否清晰
- [ ] 用户路径是否顺畅

### 2.4 框架层（Skeleton Plane）

**核心任务**：
- 界面设计：定义控件位置与交互逻辑
- 导航设计：全局/局部/上下文/友好导航
- 信息设计：视觉分组与优先级排序

**导航设计类型**：
- 全局导航：顶部菜单栏，覆盖核心功能入口
- 局部导航：页面内次级导航
- 上下文导航：内容内嵌超链接
- 友好导航：服务条款等辅助性导航

**检查清单**：
- [ ] 核心功能是否置于用户视线焦点区
- [ ] 导航结构是否清晰
- [ ] 信息优先级是否明确
- [ ] 是否保持了一致性

### 2.5 表现层（Surface Plane）

**核心任务**：
- 视觉设计：色彩策略、字体排版、图像动效
- 交互感知设计：多感官融合、情感化细节

**关键原则**：
- 视觉路径引导：用户视线应自然聚焦核心功能
- 一致性矛盾：平衡品牌统一性与场景适配
- 功能大于形式：避免过度设计干扰核心任务

**检查清单**：
- [ ] 视觉设计是否服务于功能
- [ ] 品牌识别是否清晰
- [ ] 是否通过视觉引导用户注意力
- [ ] 是否适配了不同场景

## 三、MVP策略：最小可行产品

### 核心原则

1. **验证假设**: 用最小成本验证最大假设
2. **聚焦核心**: 聚焦核心价值，避免功能冗余
3. **策略差异**: 不同产品采用不同MVP策略
4. **持续迭代**: MVP是起点，不是终点
5. **用户价值**: MVP必须提供用户价值

### MVP常见误解

- MVP ≠ 简陋版
- MVP必须验证最关键的风险假设
- MVP必须提供用户价值

### MVP策略差异

- **微信**：从"语音聊天"起步逐步扩展
- **Netflix**：从DVD租赁转型流媒体
- **滴滴**：从出租车扩展到快车/专车

### 验证重点

- [ ] 是否验证了最关键的风险假设（如用户愿付费）
- [ ] 是否提供了用户价值
- [ ] 是否聚焦了核心功能

## 四、PMF验证：产品市场匹配

### 核心原则

1. **早期信号**: 关注早期用户主动传播
2. **留存优先**: 留存率是核心指标
3. **付费验证**: 付费转化验证价值
4. **持续优化**: PMF是动态过程
5. **数据驱动**: 用数据验证PMF

### PMF三大标准

1. **早期用户主动传播**
2. **留存率 > 40%**
3. **付费转化 > 5%**

### 验证方法

- 用低成本方式验证需求
- 关注用户主动传播信号
- 监控留存率和转化率
- 持续优化产品

### 检查清单

- [ ] 是否有早期用户主动传播
- [ ] 留存率是否达到标准
- [ ] 付费转化是否达到标准
- [ ] 是否建立了数据监控体系

## 五、用户研究：特约用户与人物角色

### 核心原则

1. **真实用户**: 基于真实用户访谈，避免虚构
2. **全程参与**: 特约用户全程参与设计测试
3. **行为观察**: 观察行为 > 听用户说
4. **场景还原**: 在真实场景中验证产品
5. **持续迭代**: 用户研究贯穿产品全生命周期

### 特约用户方法

- 招募6-10名目标用户
- 全程参与设计测试
- 提供真实场景反馈
- 验证产品假设

### 人物角色（Persona）创建

- 基于真实用户访谈
- 具象化目标用户
- 避免功能泛化
- 避免虚构角色

### 检查清单

- [ ] 是否招募了特约用户
- [ ] 人物角色是否基于真实访谈
- [ ] 是否在真实场景中验证了产品
- [ ] 是否观察了用户行为

## 反模式（NEVER Do）

### ❌ 反模式1：伪需求识别

**错误做法**: 只听用户说，不观察行为
**正确做法**: 观察行为 > 听用户说
**判断标准**: 无痛点、低频次、不愿付费 = 伪需求

### ❌ 反模式2：范围蠕变

**错误做法**: 不断添加新功能
**正确做法**: 用PRD明确功能边界
**预防方法**: 严格评估每个需求的必要性

### ❌ 反模式3：过度设计

**错误做法**: 追求视觉美观，忽略功能
**正确做法**: 功能大于形式
**判断标准**: 设计是否服务于用户目标

### ❌ 反模式4：忽略验证

**错误做法**: 直接开发，不验证假设
**正确做法**: 用原型验证假设
**验证方法**: 可行性、可用性、价值三要素

### ❌ 反模式5：角色混淆

**错误做法**: 产品经理变成项目经理
**正确做法**: 产品经理定义产品，项目经理管理进度
**避免方法**: 避免陷入"监工陷阱"

## 核心金句

> "用户不为辛苦买单，只为价值投票"

> "少做功能，多问为什么"

> "观察行为 > 听用户说"

> "产品经理定义'做什么'，开发决定'怎么做'"

> "原型是可点击的假设"

> "用最小成本验证最大假设"

## 使用示例

**用户**："帮我评估一个产品机会"

**响应流程**：
1. **十问模型**：逐一回答10个关键问题
2. **检查清单**：验证用户痛点、市场规模、团队能力、时机
3. **伪需求识别**：观察行为 > 听用户说
4. **输出评估报告**：包含风险和建议
