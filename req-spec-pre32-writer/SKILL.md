# 需规生成器（1 ~ 3.1.3）

用于撰写、改写或修订中文需求规格说明书的前半部分，即 `1` 章至 `3.1.3`。本 skill 负责把“总述、背景、范围、边界、总体功能框架”写清楚，为后续 `3.2` 及后续章节提供稳定的命名、边界和结构前提。

## 先读文件

- 先读用户指定的正式模板 `.docx`。
- 再读目标需求规格说明书 `.docx`。
- 再读正文来源材料，如 PRD、调研稿、会议纪要、交付版需规、评审意见。
- 再读本 skill 目录下的 `references/需规写作宪法.md`。
- 如果用户明确要求严格排版一致，联动 `doc` 或 `docx` 负责 Word 层面的格式执行。

## 适用范围

- `1 范围`
- `1.1 标识`
- `1.2 系统概述`
- `1.3 文档概述`
- `2 引用文档`
- `3.1` 到 `3.1.3`

不要提前展开 `3.2` 的详细功能表和用例表。前文只负责把后文会用到的名称、角色、边界、主链路和总体框架交代清楚。

## 写作目标

- 让后文出现的子系统、角色、对象、接口、术语，都能在前文找到来源。
- 让全文保持“总分”关系，而不是前文泛泛而谈、后文突然出现新名词。
- 让章节结构、标题名称、编号层级与模板保持一致。
- 让前文先回答“业务总盘子是什么、系统插在哪、为什么先做这里”，再回答“系统内部有哪些模块”。

## 结构规则

- 前文先写整体，再写分层，再写边界，再写总体功能关系。
- 如果某个段落小标题承担真实分节作用，就升格为带编号的小节，不要保留无编号的悬浮标签。
- 例如 `核心功能模块边界`、`外部接口边界`、`职能边界` 这类内容，应整理为 `1.2.3.1`、`1.2.3.2`、`1.2.3.3` 之类的编号小节，而不是孤零零一行正文。
- 后文 `3.2` 要写到的子系统名称、功能总类、角色职责、数据对象，在 `系统概述` 或 `需求概述` 中必须已经出现。
- 如果项目嵌在更大的业务链路里，前文要先写“业务总层”，再写“系统当前重点赋能的环节或节点”，最后才写“系统内部模块层”。
- 不要把“业务总层”和“系统模块层”写混。前者回答系统赋能哪类业务环节，后者回答系统内部怎么分工。

## 项目背景写法

- `1.2.1 项目背景` 优先采用分层写法，而不是零散口语化说明。
- 如材料允许，优先拆成以下几类子节：
  - `1.2.1.1 行业背景`
  - `1.2.1.2 需求分析`
  - `1.2.1.3 公司背景`
  - `1.2.1.4 当前状态`
  - `1.2.1.5 建设必要性`
- 这些子节的标题名称是否保留，以模板和用户要求为准；如果模板已有固定标题，必须服从模板。
- 项目背景尽量使用完整自然段，不要机械写成 `1. 2. 3. 4. 5.`。
- 不要在项目背景里生硬写“过去怎么样、现在怎么样”的对照口号，除非用户明确要求放在该处。
- 背景段落要像正式交付文档，先讲行业与业务背景，再讲当前系统条件与建设必要性。
- 当材料足够时，优先形成以下叙事节奏：
  - 总体业务全貌
  - 业务切片与系统切入点
  - 典型场景样板
  - 平台底座、数据来源与差异化价值
- 如果系统只赋能业务链条中的审核/确认节点，要明确写出“系统不替代业务算法或业务系统本体，而是赋能关键审核节点”。

## 业务范围与软件边界

- `业务范围` 要写成可承接后文各子系统的总述，不要只列松散点。
- `软件边界` 必须把系统负责什么、不负责什么、与哪些外部系统或人工环节衔接清楚。
- 边界描述里出现的模块名、角色名、接口名，必须与后文章节保持同名。
- `业务范围` 中应显式区分：
  - 业务层：系统赋能哪些业务环节、哪些是当前重点落点、哪些是延伸环节
  - 系统层：系统内部由哪些模块承接这些落点

## 功能框架及关系描述

- `3.1.2` 只写总体功能框架和关系，不展开成 `3.2` 那种详细分解。
- 建议写出“总层 - 子系统层 - 次级分层”的稳定结构，为 `3.2.1`、`3.2.2`、`3.2.3` 铺垫。
- 若项目有清晰业务主链路，可采用“业务总层 - 系统子系统层 - 次级分层”的双层映射写法，让读者先看到业务落位，再看到系统分工。
- 如果模板中有“主要功能一览表”“功能划分一览表”等固定表，表头文字必须与模板完全一致。
- 功能名称、标识、备注列不要擅自改名。

## Word 排版规则

- 正文段落保持首行缩进。
- 正文字体、字号、行距以模板为准；如果用户已明确，则优先按用户要求执行。
- 重写或挪动正文后，要复查首行缩进是否丢失。
- 不要让普通正文携带标题级别或大纲级别，否则会污染目录。
- 不要把真正的章节标题写成普通正文再手动敲编号。前文的 `范围`、`系统概述`、`需求/GN`、`要求的状态和方式` 等都必须是 Word 真实标题。

## 目录规则

- 目录里只能出现真实标题。
- 普通正文、说明文字、图题、表题、列表项不应进入目录。
- 调整章节后要更新目录，并核查目录层级是否与模板一致。

## 与 `doc` / `docx` 的分工

- 本 skill 负责结构、命名、内容逻辑与章节承接。
- `doc` 或 `docx` 负责 Word 内的字体、段落、表格、目录、标题样式、封面等格式执行。
- 当用户强调“除了正文内容不同，其他都要跟模板一样”时，必须联动 Word skill 执行最终格式对齐。

## 自检清单

- 是否严格止步于 `3.1.3`。
- 是否没有把 `3.2` 的详细功能表提前写进前文。
- 是否所有后文关键命名都已在前文出现。
- 是否消除了无编号的悬浮小标题。
- 是否正文保持首行缩进。
- 是否目录中只保留真实标题。
- 是否前文整体结构能够自然承接后续 `3.2`、`3.3`、`3.4`。
