# 方法论/教程 (Methodology Archetype)

> 来源：7位头部AI博主分析报告综合提炼（藏师傅、饼干哥哥、卡兹克、黄叔、洛小山、赛博禅心、卡尔）

---

## 适用场景

当文章主题是以下任一情况时使用本原型：

- 分享某个工具/技术的完整使用教程
- 介绍一套自己验证过的方法论或工作流
- 手把手教读者完成某个具体任务
- 从零开始搭建某个系统/项目的全流程记录
- 总结自己踩坑后得出的最佳实践

**核心原则**：不是知识的展示，而是「一个过来人帮你把路趟平」。

---

## 结构模板（含比例分配）

```
痛点/承诺（10%）
  └─ 你的读者卡在什么地方？
  └─ 读完这篇文章他们能获得什么？

为什么这个方法有效（15%）
  └─ 方法背后的逻辑/原理
  └─ 为什么你选择这个方案而不是其他

分步拆解（50%）
  └─ 每一步的结构：
      ├─ 清晰指令（做什么）
      ├─ 截图/代码展示（长什么样）
      └─ 常见错误（会踩什么坑）
  └─ 步骤之间有清晰的逻辑递进

学习曲线坦白（10%）
  └─ 完成全流程大概需要多久？
  └─ 哪些步骤最容易卡住？
  └─ 什么前置知识是必须的？

行动清单总结（15%）
  └─ 全流程的精简版 checklist
  └─ 可复制的关键资源（prompt/代码/配置/链接）
  └─ "下一步"建议
```

---

## 开头模式（含中文示例）

### 模式一：痛点-解决方案

先戳痛点，再给希望。

> MCP的配置还是过于繁琐了，对普通人门槛有点高。这几天终于摸索出来了一些方法。
> -- 藏师傅

**要点**：痛点要具体到"这个教程要解决的那个痛点"。「终于」这个词很关键——暗示作者也经历了摸索过程。

### 模式二：结果前置

先给结果，再讲过程。让急性子读者3秒判断值不值得读。

> 我用一天时间做了个Claude Code桌面端，功能全齐、颜值在线、已经开源。
> -- 藏师傅

**要点**：结果要具体可感——时间投入+成果描述+可验证性（"已经开源"）。这就是饼干哥哥说的「结论前置」。

### 模式三：反预期

打破读者的固有认知，制造"这都行？"的好奇心。

> 6202年了，该不会还有人用AI编程开发APP吧？
> -- 饼干哥哥

**要点**：反预期开头后面必须紧跟"为什么这么说"的逻辑闭环，否则就是标题党。

## 逐节写作指导

### 第一节：痛点/承诺（约10%篇幅）

**目标**：让读者确认"这就是我要找的教程"，同时设定预期。

写法要求：
- 描述一个具体的卡点（不是"想学XX"，而是"在XX这步卡住了"）
- 给出明确承诺："读完这篇，你能做到XX"
- 如果有前置条件，提前说清楚（"你需要先有XX基础"）
- 可选：用「结论前置」技巧——在前3段给出 TL;DR 总结

**禁忌**：不要承诺"超简单""零基础也能学会"，除非真的是。

### 第二节：为什么这个方法有效（约15%篇幅）

**目标**：让读者理解方法背后的逻辑，而不只是机械跟步骤。

写法要求：
- 解释为什么选择这个方案（"我试过XX和YY，最终选了ZZ因为..."）
- 如果涉及技术原理，用类比解释（不是论文式讲解）
- 说清楚这个方法的适用边界（"这个方法适合XX场景，如果你的情况是YY，可能需要..."）
- 可选：简单对比其他方案的优劣

### 第三节：分步拆解（约50%篇幅 -- 核心部分）

**目标**：让读者跟着操作就能完成，同时避开常见的坑。

**每一步的标准结构**：

```
## 步骤 X：[这一步要做什么]

[1-2句说明为什么要做这一步]

[具体操作指令]

[图片：步骤X - 描述] 或 [代码/配置/命令]

> 注意：[这一步容易犯的错误/常见问题]
> 如果遇到XX错误，检查一下YY
```

写法要求：
- 步骤编号清晰（01/02/03 或 Step 1/2/3）
- 每步三件套：指令 + 截图/代码 + 常见错误
- 可复制的内容用代码块包裹
- 步骤之间有逻辑递进，不是割裂的
- 穿插个人感受（"这步我当时卡了半小时""做到这里你会发现..."）
- 重要提示用引用块（>）或加粗标注

**节奏控制**：
- 前面的步骤写详细一些（读者刚开始，需要手把手）
- 中间步骤可以适当加快节奏
- 关键难点步骤重新放慢，多给截图和错误排查

### 第四节：学习曲线坦白（约10%篇幅）

**目标**：给读者真实的预期管理。

写法要求：
- 完成全流程的时间预期（"大概需要X小时"）
- 最容易卡住的地方（"第X步可能会卡住，多试几次"）
- 前置知识要求（"需要大概了解XX"）
- 可选：推荐学习资源

**关键技巧**（卡兹克的「谦逊框架」）：

> 说实话我也不确定这个方法是不是最优解，可能有些想法还不成熟，但至少这是我实际跑通了的。

这种坦诚反而增加可信度。读者信任"我跑通了但可能不是最优"，远胜于"这是最好的方法"。

### 第五节：行动清单总结（约15%篇幅）

**目标**：给一个可以收藏、反复查看的精简版。

写法要求：
- 全流程 checklist（每步一行，可打勾）
- 所有可复制的资源集中放在这里（prompt、代码、配置、链接）
- "完成以上步骤后，你应该能看到XX"的验证标准
- "下一步"建议（"如果想进一步，可以..."）

## 必备元素清单

| 元素 | 说明 | 必要性 |
|------|------|--------|
| 步骤编号 | 01/02/03 或 Step 1/2/3 | 必须 |
| 每步三件套 | 指令 + 截图/代码 + 常见错误 | 必须 |
| 可复制资源 | prompt/代码/配置/链接 | 必须 |
| 学习曲线坦白 | 时间预期 + 难点标注 | 必须 |

| 行动清单 | 全流程的精简 checklist | 必须 |
| 结论前置 | 前3段内给 TL;DR | 推荐 |
| 前置条件说明 | 需要什么基础/工具 | 推荐 |
| 验证标准 | "做完应该能看到XX" | 推荐 |

## 反面模式（Anti-patterns）

### 1. 步骤没有配图/代码

**错误示范**：
> 接下来打开设置，找到XX选项，修改一下就好了。

**问题**：读者看不到操作界面，不知道"设置在哪""XX选项长什么样"。

### 2. 没有时间估计

**错误示范**：
（通篇教程，不告诉读者总共需要多少时间）

**问题**：读者不知道自己是在开始一个30分钟的任务还是3天的项目。

### 3. 号称"超简单"但实际不简单

**错误示范**：
> 这个方法超简单！只需要XX步就能搞定！
> （然后第3步就需要配置Linux环境变量）

**问题**：对技术栈的假设不同。对你来说"超简单"可能是因为你已经有了环境。

### 4. 没有覆盖失败模式

**错误示范**：
（只写了正确路径，没有写任何"如果出错了怎么办"）

**问题**：现实中读者大概率会遇到问题。一个不提错误处理的教程，在第一个报错处就失去了读者。

### 5. 纯理论没有实操

**错误示范**：
> AI Agent的核心在于规划能力、工具使用能力和记忆能力。规划能力是指...

**问题**：方法论文章的价值在于"我怎么做到的"，不在于"理论上应该怎么做"。

## 核心写作技巧

### 饼干哥哥的「结论前置」

在前3段内给出 "too long didn't read" 总结。让急性子读者3秒判断值不值得读，让有耐心的读者带着全局观往下看。

格式参考：
```
[痛点/场景一句话]
[核心结论/成果一句话]
[这篇文章会教你什么一句话]

下面是详细步骤 ↓
```

### 卡兹克的「谦逊框架」

开头用 disclaimer 降低读者预期，反而增加信任感。

> 说实话我也不确定这个方法是不是最优解，可能有些想法还不成熟，但至少这是我实际跑通了的。

**适用场景**：新技术、探索性方案、个人方法论。
**不适用场景**：已经非常成熟的标准教程。

### 失败模式前置

在每一步中，把常见错误写在正确操作后面，用引用块或加粗标注。这比把所有FAQ放在文末有效得多，因为读者是"遇到问题的时候"需要帮助，不是"读完整篇"以后。

## 评分示例段落

### 示例一：高分步骤示范（9/10）

来源：饼干哥哥

> 步骤一：构建文件结构
> 在你的目录下，建立如下结构：
> ~/.openclaw/
> ├── openclaw.json
> ├── skills/

**评分理由**：
- 步骤编号清晰
- "在你的目录下"—— 直接对读者说话，不是第三人称描述
- 文件结构用代码块展示，可直接复制参考
- 结构清晰，读者一眼就知道要建什么
- 扣分点：如果能加上"如果目录不存在就先 mkdir"会更完善

### 示例二：高分谦逊开场（9/10）

来源：卡兹克

> 说实话我也不确定这个方法是不是最优解，可能有些想法还不成熟，但至少这是我实际跑通了的。

**评分理由**：
- 「不确定是不是最优解」—— 诚实降低预期，反而增加信任
- 「可能有些想法还不成熟」—— 坦诚，但不是自我否定
- 「至少这是我实际跑通了的」—— 关键信息——这不是理论，是实操验证过的
- 整句话的逻辑：不完美，但真实有效
- 这比"分享一个超好用的方法"可信度高10倍

### 示例三：反面教材（3/10）

> AI自动化工作流搭建非常简单。首先你需要了解一下基本概念。AI Agent是一种能够自主完成任务的系统，它包含三个核心能力：规划、工具使用和记忆。规划能力是指......

**扣分理由**：
- 「非常简单」—— 对谁简单？这是一个需要证明的判断，不是前提
- 直接进入理论概念讲解，没有痛点切入
- 没有任何个人经验证据
- 没有步骤结构
- 读了两段还不知道具体要做什么
- 教科书语气而不是"过来人"语气

## 写作节奏参考

| 段落位置 | 情绪节奏 | 信息密度 |
|----------|----------|----------|
| 开头/承诺 | 共鸣/期待 | 低（建立动力） |
| 方法逻辑 | 理解/信服 | 中（建立认知） |
| 前几步 | 跟随/安心 | 高（手把手详细） |
| 中间步骤 | 节奏加快/自信 | 高（读者已上手） |
| 难点步骤 | 放慢/耐心 | 高（重新详细） |
| 学习曲线 | 坦诚/鼓励 | 低（情绪调节） |
| 行动清单 | 清晰/可操作 | 高（精华浓缩） |
