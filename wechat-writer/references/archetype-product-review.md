# 产品测评/工具介绍 (Product Review Archetype)

> 来源：7位头部AI博主分析报告综合提炼（黄叔、饼干哥哥、藏师傅、卡兹克、洛小山、赛博禅心、卡尔）

---

## 适用场景

当文章主题是以下任一情况时使用本原型：

- 测评某个具体 AI 产品/工具/平台
- 介绍一个新发布的软件或服务
- 对比多个同类工具并给出推荐
- 分享某个工具的实操使用经验
- 从个人视角评价某个产品的优劣

**核心原则**：不是功能说明书，而是「一个真实用户的使用判决」。

---

## 结构模板（含比例分配）

```
痛点/场景 Setup（10%）
  └─ 读者为什么需要关心这个产品？建立共鸣场景

产品介绍（20%）
  └─ 产品是什么？核心定位？谁做的？
  └─ 一句话说清楚它解决什么问题

实操 Walkthrough with Screenshots（40%）
  └─ 最核心的使用流程
  └─ 每个关键步骤配截图占位 [图片：xxx]
  └─ 穿插个人体验感受（不是客观描述，是主观判断）

与替代方案对比（15%）
  └─ 至少一个对比锚点（vs 什么？）
  └─ 核心差异用表格或并列段落呈现

判决 + 适用人群（15%）
  └─ 明确推荐/不推荐/看情况
  └─ 适合谁？不适合谁？
  └─ 使用建议（怎么上手最快）
```

---

## 开头模式（含中文示例）

### 模式一：场景/痛点切入

从读者日常遇到的具体问题出发，建立"你是不是也这样"的共鸣。

> 你是不是也有这样的烦恼？微信读书里积累了大量精彩划线，却总是躺在那里吃灰。
> -- 黄叔

**要点**：痛点必须具体到场景，不能是抽象的"效率低"。

### 模式二：直接经验切入

用第一人称的真实经历开场，制造"刚体验完热乎着告诉你"的现场感。

> 昨天参加了Fellou的发布会，今天详细体验了一下。
> -- 黄叔

**要点**：交代时间、场合，建立信任感。

### 模式三：Before/After 对比

用一组强烈反差开场，让读者直接看到产品的价值。

> 以前需要3个运营盯着的活，现在一套n8n工作流就在服务器上静悄悄地跑完了。
> -- 饼干哥哥

**要点**：对比要有具体数字或画面，不能只是"变快了""变好了"。

## 逐节写作指导

### 第一节：痛点/场景 Setup（约10%篇幅）

**目标**：让读者在3秒内确认"这篇文章跟我有关"。

写法要求：
- 用「你」开头或在前两句内出现「你」
- 描述一个具体的微场景（不是宏观趋势）
- 痛点要精准到"这个工具能解决的那个痛点"
- 如果是新品类，先花1-2句解释这个品类存在的意义

**禁忌**：不要用"随着XX的发展..."开头。

### 第二节：产品介绍（约20%篇幅）

**目标**：让读者快速理解产品定位和核心卖点。

写法要求：
- 产品全名 + 版本号（如有）
- 一句话定位："XX 是一个帮你做 YY 的 ZZ"
- 创始团队/背景（如果有加分信息）
- 核心功能不超过3个要点
- 个人使用时长/深度交代（"我用了一个月""实测3天"）

**禁忌**：不要照搬官网介绍，用自己的话重新表述。

### 第三节：实操 Walkthrough（约40%篇幅 -- 核心部分）

**目标**：让读者跟着文章就能上手，同时感受到作者的真实判断。

写法要求：
- 按使用流程组织，不按功能列表组织
- 每个关键步骤：操作说明 + 截图占位 + 个人感受
- 截图占位格式：`[图片：步骤X - 描述]`
- 穿插"wow moment"（用卡兹克的「升番」技巧：最惊艳的体验留到最后）
- 对不好用的地方直接吐槽
- 包含至少一个"我踩过的坑"

**节奏控制**：
- 前1/3正常介绍，建立基本认知
- 中1/3深入细节，展示专业度
- 后1/3放大招（最惊艳的功能/最意外的发现）

### 第四节：与替代方案对比（约15%篇幅）

**目标**：帮读者在已有选项中定位这个产品。

写法要求：
- 至少一个明确的对比对象（"相比 XX"）
- 核心差异维度不超过5个
- 每个维度给出明确判断（谁更好，好多少）
- 适合用表格呈现的就用表格
- 诚实面对竞品优势（"这方面确实XX更强"）

**禁忌**：不要假装竞品不存在。

### 第五节：判决 + 适用人群（约15%篇幅）

**目标**：给出一个清晰的、可操作的结论。

写法要求：
- 三选一明确判决：推荐 / 不推荐 / 看情况
- 如果是"看情况"，必须说清什么情况推荐、什么情况不推荐
- 适用人群画像（2-3类人）
- 不适用人群画像（1-2类人）
- 上手建议（先做什么再做什么）

## 必备元素清单

| 元素 | 说明 | 必要性 |
|------|------|--------|
| 产品名 + 版本号 | 具体到版本，不能只说"某AI工具" | 必须 |
| 个人使用证据 | "我用了X天""实测X个功能" | 必须 |
| 截图占位 | `[图片：xxx]` 格式 | 必须 |
| Wow moment | 至少一个"这个功能惊到我了"的瞬间 | 必须 |
| 诚实局限 | "不过有个问题""说实话这块还不太行" | 必须 |
| 明确判决 | 推荐/不推荐/看情况 | 必须 |
| 对比锚点 | vs 什么替代方案 | 必须 |
| 适用人群 | 适合谁/不适合谁 | 必须 |
| 价格/成本信息 | 免费/付费/定价模式 | 推荐 |
| 上手时间预期 | "大概需要X分钟/小时上手" | 推荐 |

## 反面模式（Anti-patterns）

### 1. 功能清单式写作
**错误示范**：
> 产品支持XX功能、YY功能、ZZ功能。此外还支持...

**问题**：读者看的是测评不是说明书，需要的是判断而非罗列。

### 2. 没有个人使用证据
**错误示范**：
> 据了解，这款产品可以实现XX...

**问题**：失去测评的核心价值——真实体验。读者为什么不直接看官网？

### 3. 没有对比锚点
**错误示范**：
> 这款产品非常好用，功能全面。

**问题**：好用是相对于什么好用？全面是相对于什么全面？缺少参照系。

### 4. 空洞赞美
**错误示范**：
> 这款产品非常强大，功能非常全面，体验非常流畅。

**问题**：三个"非常"等于零个信息。强大在哪？全面到什么程度？流畅的标准是什么？

### 5. 回避缺点
**错误示范**：
（通篇只说优点，完全不提任何不足）

**问题**：读者会觉得这是软文而非测评，丧失信任。

## 评分示例段落

### 示例一：高分开头（9/10）

来源：饼干哥哥

> 废话不多说，先说结论。最近这半个月，我把团队里的营销SOP重新梳理了一遍。结果非常惊人。以前需要3个运营盯着的活，现在一套n8n工作流就在服务器上静悄悄地跑完了。

**评分理由**：
- 「废话不多说，先说结论」—— 结论前置，尊重读者时间
- 「最近这半个月」—— 具体时间，证明亲测
- 「3个运营→一套工作流」—— 具象的 Before/After 对比
- 「静悄悄地跑完了」—— 画面感，不是抽象的"提升效率"

### 示例二：高分 Walkthrough（8/10）

来源：藏师傅

> 步骤一：构建文件结构。在你的目录下，建立如下结构...

**评分理由**：
- 编号清晰，可跟随操作
- "在你的目录下"—— 直接对读者说话
- 配合代码/结构展示，可直接复制
- 扣分点：如果能加上"这步可能遇到的坑"会更好

### 示例三：高分判决（9/10）

来源：洛小山

> 选工具的本质不是选功能，是选你跟AI的相处方式。

**评分理由**：
- 一句话升维：从功能对比升维到使用哲学
- 帮读者建立了选择框架，而不只是给了一个答案
- 有记忆点，读完能复述
- 这才是测评文章判决部分的最高水平：不只是"推荐"，而是给你一个判断标准

## 写作节奏参考

| 段落位置 | 情绪节奏 | 信息密度 |
|----------|----------|----------|
| 开头 | 共鸣/好奇 | 低（留白引读者往下看） |
| 产品介绍 | 客观/信任 | 中（快速建立认知） |
| Walkthrough 前半 | 平稳/专业 | 高（干货输出） |
| Walkthrough 后半 | 惊喜/兴奋 | 高（wow moment 释放） |
| 对比 | 理性/公正 | 中（结构化呈现） |
| 判决 | 果断/有态度 | 低（结论清晰不啰嗦） |
