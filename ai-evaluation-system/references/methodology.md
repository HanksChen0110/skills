# AI产品评估体系完整方法论

> 来源：[[AI产品评估体系]]
> 工作流位置：需求挖掘 → 用户调研 → 需求分析 → 竞品分析 → 技术选型 → 数据处理 → AI PRD → **评估体系** → Badcase治理

---

## 一、为什么 AI 产品必须建立评估体系

AI 产品不同于传统软件：功能对不对可以测试，但**AI 的输出质量是概率性的**，必须用评估体系来量化"好不好"。

> **核心目标**：将业务目标转化为可量化的测试集和评分标准，才能比较调优效果，最终指导业务目标的达成。

**没有评估体系会怎样？**
```
- 不知道模型到底好不好（凭感觉）
- 优化后无法证明有没有提升
- 无法向老板/业务方汇报进展
- badcase 修复后可能在其他地方引入新问题
```

## 二、评估体系全景

```
测试数据集（评测集）
    ↓
评估指标体系（三层）
    ↓
评估方法
  ├── 基于规则（RAGAS 框架）
  ├── 大模型评估（LLM-as-Judge）
  └── 人工评估
    ↓
结果分析 → 优化方向 → Badcase治理
```

## 三、评测集构建

### 3.1 评测集来源

| 来源方式 | 说明 | 优缺点 |
|---------|------|--------|
| **基于历史用户问题** | 从线上真实 query 中抽取 | 最贴近真实场景，质量参差不齐 |
| **人工构造（Manually Curated）** | 由业务专家设计典型问题 | 质量高，但成本高、覆盖有限 |
| **大模型批量生成（Synthetic）** | 用 LLM 生成多样化测试用例 | 覆盖广、成本低，但需人工审核质量 |
| **从执行日志中提取** | 线上日志挖掘高频 / 失败 case | 针对性强，适合 badcase 专项测试 |

### 3.2 评测集数据格式

```python
data_samples = {
    'question': ['用户提问1', '用户提问2'],
    'answer': ['模型给出的答案1', '模型给出的答案2'],
    'contexts': [
        ['RAG检索到的上下文段落1', '上下文段落2'],
        ['RAG检索到的上下文段落3']
    ],
    'ground_truth': ['标准答案1', '标准答案2']
}
```

### 3.3 评测集质量要求

```
数量要求：
  - 最低 50 条（才有统计意义）
  - 推荐 100-200 条
  - 抽样必须随机

质量检验：
  - 自己标注 100 条，与预期对比
  - 准确率 ≥ 95% → 可靠
  - 准确率 < 90% → 先修复评测集质量

注意：开源评测集（如 LCQMC）也可能有标注错误
```

## 四、三层指标体系

### 4.1 顶层：业务指标

```
- 用户留存率（Day1 / Day7 / Day30）
- 付费转化率
- DAU / MAU（活跃度）
- 人工替代率（AI 替代人工的比例）
- NPS（净推荐值）
```

### 4.2 中层：产品体验指标

**任务型产品**：
```
- 任务完成率：用户发起的任务中，AI 成功完成的比例
- 独立解决率：不需要转人工直接解决的比例
- 转人工率：AI 无法处理，转给人工的比例
- 平均响应时长
```

**内容型产品**：
```
- 用户采纳率：AI 生成内容被用户接受/使用的比例
- 内容质量评分（用户主动评分）
- 分享率
```

**通用指标**：
```
- CSAT（用户满意度评分，1-5 分）
- 用户点踩率（负反馈率）
```

### 4.3 底层：模型效果指标

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **准确率（Precision）** | 检索/预测正确的比例 | 正确预测数 / 总预测数 |
| **召回率（Recall）** | 找到了多少相关内容 | 检索到的相关数 / 总相关数 |
| **F1 Score** | 准召综合指标 | 2×P×R / (P+R) |
| **忠实度（Faithfulness）** | 回答是否基于检索到的文档，有无幻觉 | 可由文档推断的声明数 / 总声明数 |
| **答案相关性** | 答案是否回答了用户的问题 | 基于嵌入相似度计算 |
| **上下文相关性** | 检索到的 chunk 中有多少与问题相关 | 相关句数 / 检索句总数 |
| **上下文召回率** | 检索结果与标准答案的覆盖程度 | - |
| **安全红线触发率** | 触发有害内容的频率 | 触发次数 / 总请求数 |

## 五、评估方法

### 5.1 RAGAS 框架（RAG 专用）

RAGAS 是专门用于 RAG 系统的自动评估框架，**无需人工标注即可运行**。

**核心评估维度：**

```
忠实度（Faithfulness）：
  → 把答案拆分为多个声明
  → 判断每个声明是否可从 contexts 中推断出来
  → 忠实度 = 可推断声明数 / 总声明数

答案相关性（Answer Relevancy）：
  → 基于答案推测出多个问题
  → 计算用户原问题与推测问题的嵌入相似度
  → 取平均值

上下文精确率（Context Precision）：
  → 检索结果中有多少与答案相关

上下文召回率（Context Recall）：
  → 检索结果与标准答案的覆盖程度

答案正确性（Answer Correctness）：
  → 与 ground_truth 对比，答案是否正确
```

### 5.2 LangSmith 评估

```
使用 LangSmith 构建评估 Pipeline：
  a. 数据集 Dataset：手动构造 / 日志提取 / LLM 生成
  b. 评估器 Evaluator：
     - LLM-as-Judge（大模型打分）
     - Heuristics（启发式规则）
     - Human（人工评价）
  c. 评估模式 Mode：
     - Comparison：对比两个方案
     - Reference-free：无标准答案时的相对评估
     - Ground Truth：与标准答案对比
```

### 5.3 大模型评估（LLM-as-Judge）

**评分标准设计示例（5 分制）：**

| 分数 | 评分标准 |
|------|---------|
| 0 分 | 无结果 |
| 1 分 | 有结果，但文不对题，没有理解问题 |
| 2 分 | 有结果，理解了问题，但有大量无关信息，存在误导性 |
| 3 分 | 有结果，理解了问题，但回答不够全面，仅回答出部分要点 |
| 4 分 | 有结果，理解了问题，回答较全面，能回答出约 80% 要点 |
| 5 分 | 有结果，理解了问题，回答全面，无不相关内容，言简意赅 |

**Prompt 模板（批量评测）：**
```markdown
你是 AI 数据评测和打分专家。现在有一批在大模型测试集上跑出来的答案，需要你对答案进行评分。

## 评分标准
[见上表]

## 测试数据样例
### 给大模型的问题
[问题内容]

### 参考答案
[标准答案]

### 大模型的回答
[待评分答案]

请给出 0-5 分的评分，并简要说明理由。
```

### 5.4 人工评估

虽然不方便，但可靠性不低。适用于**高风险场景**或**LLM 评估不准确**时的兜底方案。

**评估维度：**
```
- 真实性：回复内容是否真实、完整、无幻觉、无过度描述
- 相关性：答案是否有效回答了问题
- 无害性：内容是否安全合规，无有害信息
- 拒识能力：没有答案时，是否恰当地拒绝回答
```

**评分方式**：简单的 0/1 打分，或 0/0.5/1 三档，保持一致性即可。

## 六、A/B 测试

### 6.1 何时做 A/B 测试

```
- 对比两种 Prompt 策略的效果
- 对比不同模型的表现
- 对比 RAG 参数调整前后的效果
- 评估新功能对核心指标的影响
```

### 6.2 A/B 测试设计原则

```
1. 单变量原则：每次只改变一个变量
2. 随机分流：用户随机分配到 A/B 组
3. 足够样本量：确保统计显著性（通常需要 p < 0.05）
4. 明确成功标准：提前定义"什么算赢"
5. 设置保护机制：如 A 组效果显著差，快速切回
```

## 七、MVP 哲学与评估

### 7.1 MAP（最小可接受性能）

```
AI 产品的 MVP 核心不再是"最小功能集"，而是：
  → 最小可接受性能（MAP: Minimum Acceptable Performance）

定义 MAP 的方法：
  1. 调研用户对"失败回答"的容忍底线
  2. 设定业务指标的最低阈值（如：满意度 ≥ 3.5 / 5.0）
  3. 设定模型指标的最低阈值（如：准确率 ≥ 85%）
  4. 低于 MAP → 需要人机协作兜底，不能全自动上线
```

### 7.2 HITL（Human-in-the-Loop）人机协作

```
当 AI 置信度低时，触发人工审核：
  - 设置置信度阈值，低于阈值 → 转人工
  - 人工处理结果反哺训练数据
  - 随着数据积累，逐步降低转人工的频率
```

## 八、评估结果的应用

```
评估结果 → 定位问题类型 → 选择优化方向

问题：忠实度低（有幻觉）
  → 优化 RAG 检索质量 / 加强 Prompt 约束

问题：答案相关性低（答非所问）
  → 优化意图识别 / 改善 Query 理解

问题：上下文召回率低（找不到）
  → 优化分块策略 / 更换 Embedding 模型

问题：准确率高但召回率低（过于保守）
  → 调低置信度阈值 / 扩展知识库覆盖
```

## 九、面试常见问题

**Q：AI产品怎么评估效果？**
> 回答框架：评测集构建 → 三层指标体系（业务/产品/模型）→ 评估方法选择（RAGAS/LLM-as-Judge/人工）→ A/B测试验证 → 结果应用

**Q：RAGAS框架是什么？**
> 回答框架：RAGAS是RAG系统自动评估框架，评估忠实度、答案相关性、上下文精确率/召回率等维度，无需人工标注

**Q：MAP是什么意思？**
> 回答框架：Minimum Acceptable Performance，AI产品MVP的核心是最小可接受性能而非最小功能集。低于MAP需要人机协作兜底
