# AI-PRD撰写检查清单

## 一、产品分类检查

### 嵌入型AI产品设计要点
- [ ] 把 AI 要完成的任务说得很具体
- [ ] 写明白AI它需要哪些数据
- [ ] 需要做一些兜底的策略

### Agent型AI产品设计要点
- [ ] 想明白关键问题：这个 Agent 在业务里的位置究竟是什么？
- [ ] 文档里需要把几件事说白：
  - [ ] 它对什么问题有回答义务
  - [ ] 对什么问题只有建议权
  - [ ] 什么操作必须经过用户或人工审核
  - [ ] 遇到灰区时优先遵守哪条原则
- [ ] 当你习惯在 PRD 里写"权限、义务、边界、路径"这类内容，Agent 的行为才有机会在上线后长期保持稳定

## 二、核心结构完整性检查

### 最小可行PRD（MVP阶段）
- [ ] 第一部分：产品目标与范围（含能力边界）
- [ ] 第三部分：核心工作流与意图定义
- [ ] 第七部分：异常处理与兜底策略（异常情况处理表）

### 标准PRD（正式开发阶段）
- [ ] 第一部分：产品目标与范围（含能力边界）
- [ ] 第二部分：Agent核心人设（角色定位、语言风格、行为准则、行为禁忌）
- [ ] 第三部分：核心工作流与意图定义
- [ ] 第四部分：技能与工具集详述（功能、输入/输出、调用时机）
- [ ] 第七部分：异常处理与兜底策略（异常情况处理表）

### 完整PRD（成熟产品迭代）
- [ ] 第一部分：产品目标与范围（含能力边界）
- [ ] 第二部分：Agent核心人设（角色定位、语言风格、行为准则、行为禁忌）
- [ ] 第三部分：核心工作流与意图定义
- [ ] 第四部分：技能与工具集详述（功能、输入/输出、调用时机）
- [ ] 第五部分：知识库与RAG策略（来源、检索优先级、冲突处理、引用溯源）
- [ ] 第六部分：对话逻辑与COT应用（思考链步骤）
- [ ] 第七部分：异常处理与兜底策略（异常情况处理表）

## 三、商业与数据思维检查

- [ ] 成本评估（Token成本估算）
- [ ] Build vs Buy决策
- [ ] 三层指标体系定义（业务/体验/模型）
- [ ] MAP（最小可接受性能）定义
- [ ] 人机协作(HITL)触发点设计

## 四、PRD质量自检清单

### 逻辑一致性检查
- [ ] 产品目标与Agent人设是否一致？
- [ ] 工作流设计是否支持产品目标？
- [ ] 工具集是否覆盖了工作流的所有需求？
- [ ] 异常处理是否覆盖了所有可能的失败点？

### 可执行性检查
- [ ] 研发能否根据PRD直接开始开发？
- [ ] 每个工具的参数定义是否足够清晰？
- [ ] 异常处理的兜底策略是否可操作？
- [ ] 所有关键决策点是否都有明确的判断标准？

### 可验证性检查
- [ ] 是否定义了明确的成功指标？
- [ ] 是否设计了验证方法（A/B测试、用户反馈等）？
- [ ] MAP（最小可接受性能）是否可量化？
- [ ] 评测体系是否完整（业务/体验/模型三层指标）？

### 风险控制检查
- [ ] 是否识别了主要风险点？
- [ ] 是否设计了风险应对策略？
- [ ] 是否考虑了成本超支的可能性？
- [ ] 是否明确了适用边界（什么时候不应该用AI）？

### 不确定性处理检查
- [ ] 是否明确识别了确定性场景（必须给确定答案）？
- [ ] 是否明确识别了建议性场景（可以给相对合理建议）？
- [ ] 是否明确识别了交互性场景（必须先问清楚再行动）？
- [ ] 每个场景的兜底策略是否清晰？

## 五、内容质量检查

### 目标描述质量
- [ ] 目标是否具体、可量化？
- [ ] 核心价值是否清晰（区别于传统方案）？
- [ ] 范围边界是否明确（什么能做，什么不能做）？

### Agent人设质量
- [ ] 角色定位是否具体（不是"友好、热情、专业"这种泛泛描述）？
- [ ] 语言风格是否有具体示例？
- [ ] 行为准则是否可操作？
- [ ] 行为禁忌是否明确？

### 工作流质量
- [ ] 核心意图(Intents)是否完整？
- [ ] 每个意图的响应流程是否清晰？
- [ ] 工作流是否支持产品目标？

### 工具集质量
- [ ] 每个工具的功能描述是否清晰？
- [ ] 输入/输出参数是否完整？
- [ ] 调用时机是否明确？

### 异常处理质量
- [ ] 是否制作了异常情况处理表？
- [ ] 常见异常情况是否都覆盖？
- [ ] 每个异常情况的兜底策略是否可操作？

## 六、认知偏差检查

### 避免过度自信
- [ ] 是否说明了PRD的局限性？
- [ ] 是否避免了"写了PRD就能成功"的假设？
- [ ] 是否考虑了市场、技术、团队等外部因素？

### 避免规划谬误
- [ ] 是否参考了类似项目的历史数据？
- [ ] 时间估算是否考虑了缓冲？
- [ ] 是否考虑了技术实现的复杂度？

### 避免确认偏误
- [ ] 是否主动考虑了失败场景？
- [ ] 是否考虑了反例（什么时候不适用）？
- [ ] 是否邀请了外部视角参与评审？

## 七、快速检查清单（5分钟版本）

如果时间有限，至少检查这5项：

1. [ ] **目标是否清晰**：产品目标、核心价值、范围边界是否明确？
2. [ ] **边界是否定义**：什么能做、什么不能做、什么时候不适用？
3. [ ] **异常是否处理**：是否制作了异常情况处理表？
4. [ ] **指标是否定义**：是否定义了MAP和三层指标体系？
5. [ ] **风险是否识别**：是否识别了主要风险点和应对策略？

---

**记住**：PRD是起点，不是终点。持续迭代，持续验证。
