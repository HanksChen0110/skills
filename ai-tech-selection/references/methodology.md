# AI产品技术选型完整方法论

> 来源：[[AI产品技术选型]]
> 工作流位置：需求挖掘 → 用户调研 → 需求分析 → 竞品分析 → **技术选型** → 数据处理 → AI PRD → 评估体系 → Badcase治理

---

## 一、核心思想

> **大模型本质还是工具，和传统 NLP、规则匹配等是平权的。根据场景需求因地制宜，不要为了用大模型而用大模型——杀鸡焉用牛刀。**

技术选型不是"选最先进的"，而是"选最合适的"。

---

## 二、第一步：判断是否需要 AI / 大模型

### 2.1 与传统技术的对比

| 对比维度 | 传统 NLP / 规则 | 大模型 |
|---------|--------------|--------|
| **知识覆盖** | 需人工整理，可扩展性差 | 海量开放领域知识 |
| **泛化能力** | 差，需大量规则维护 | 强，能处理长尾场景 |
| **可控性** | 高，结果可预测 | 低，存在幻觉风险 |
| **数据安全** | 可完全私有化 | 外部 API 有泄露风险 |
| **成本** | 低（规则是一次性的） | 高（按 Token 持续计费） |
| **时效性** | 人工更新，灵活 | 训练数据有截止日期 |

### 2.2 什么时候适合用大模型？

评估以下 7 个维度：

```
1. 硬件基础：是否有足够的算力资源（GPU/服务器）？
2. 技术基础：当前任务需要哪种技术选型？能达到需求粒度吗？
3. 数据获取：任务中的数据可获取吗？量足够吗？质量高吗？
4. 任务泛化性：任务边界是否明确？泛化性是否强？
5. 错误容忍度：这个任务允许出错吗？是否有人工审查环节？
6. 可解释性：需要解释回答是如何生成的吗？
7. 应答时效性：对响应时延的要求是多少？业务能接受多长延迟？
```

> **一句话判断**：高泛化、数据充足、有容错机制 → 适合用大模型；强确定性、低容错、成本敏感 → 优先考虑规则/传统方案

## 三、第二步：选择 AI 架构方案

### 3.1 四种方案对比

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **直接调用 API** | 使用 OpenAI / Claude / Deepseek 等外部 API | 快速验证、非敏感数据、预算有限 |
| **RAG（检索增强）** | 大模型 + 私有知识库检索 | 知识时效性要求高、需要引用源头 |
| **微调（Fine-tuning）** | 在基础模型上用私有数据训练 | 有大量标注数据、需要特定风格/领域深度 |
| **Agent / 工作流** | 多工具协作、多步推理 | 复杂任务自动化、跨系统操作 |

### 3.2 RAG vs 微调的选择

```
优先选 RAG 的场景：
  - 知识需要频繁更新（文档库、法规库）
  - 需要追溯答案来源（可信度要求高）
  - 数据量不足以微调
  - 快速上线，验证 MVP

优先选微调的场景：
  - 需要特定语言风格或格式
  - 有大量高质量标注数据（>1000 条）
  - 通用模型在该垂直领域表现不够好
  - 需要降低推理时的 Prompt 长度（节省 Token）
```

## 四、第三步：技术架构选型（AI 沙盘模型）

### 4.1 四大维度

**① 基础设施（算力来源）**
```
- 云端 API 调用：按 Token 计费，无需硬件投入，适合初期
- 本地私有化部署：需要 GPU 集群，适合数据安全要求高的场景
- 混合部署：敏感数据本地，通用功能走云端

关键考量：并发量需求、响应速度、能耗成本（本地部署时）
```

**② 模型选择**
```
核心思想：选择大模型最重要的是验证"AI 擅不擅长做你想让它做的事"

按能力选：
  - 长文本理解  → Kimi K2
  - 深度推理    → Deepseek / 豆包
  - 代码能力    → Claude / GPT-4
  - 文生图      → SDXL / Midjourney / Seedream
  - 视频生成    → Sora / 即梦
  - 本地化需求  → Ollama 本地部署开源模型
```

**③ 数据策略**
```
- 数据来源：企业知识库 / 实时搜索 / 结构化数据库
- 数据清洗：质量控制（见数据处理模块）
- RAG 配置：分块策略、Embedding 模型、检索方式
```

**④ 业务流程（应用层）**
```
- 单次对话：直接 LLM 调用
- 多轮对话：维护上下文管理
- 复杂任务：Agent（ReAct / Plan-and-Execute）
- 跨系统操作：MCP 接入
- 使用方式：原生产品 / API 接入 / MCP 被接入 / SDK
```

## 五、第四步：模型具体选型

### 5.1 从公司属性出发

```
- 政府投资企业      → 优先选国产模型（Deepseek / 文心 / 通义）
- 外资基因企业      → 可选 GPT / Claude 等国际模型
- BAT 系公司       → 天然使用对应生态模型（阿里→通义 / 腾讯→混元 / 百度→文心）
```

### 5.2 从模型业务能力出发

**查看评测网站：**
```
- OpenCompass 司南（上海 AI Lab）：全面评测平台
- SuperCLUE（推荐）：中文通用大模型综合性基准
- ModelScope 魔搭社区：Eval-Scope 评估工具
- HuggingFace MTEB：Embedding 模型排行榜
```

### 5.3 从成本出发

```
以通义千问（Qwen）系列为例，模型参数量越大，能力越强，Token 费用越高：
  Qwen2.5: 0.5B / 1.5B / 3B / 7B / 14B / 32B / 72B

选型原则：找到能满足需求的最小参数量模型
  → 用评测集验证效果 → 对比 Token 成本 → 做决策
```

### 5.4 从版权出发

```
⚠️ 必须确认模型授权协议允许商业用途
  - MIT / Apache 2.0：可商用
  - CC BY-NC：仅限非商业使用
  - 需仔细阅读 License，避免后续版权纠纷
```

## 六、Build vs Buy 决策框架

| 维度 | Buy（调用外部 API） | Build（自研/微调） |
|------|-----------------|----------------|
| **优点** | 快速上线、低门槛、无需硬件 | 数据安全、形成长期壁垒、可定制 |
| **缺点** | 数据隐私风险、长期成本高、无核心壁垒 | 成本高、周期长、技术门槛高 |
| **适合** | 初期验证、非敏感场景、预算有限 | 数据敏感、有大量数据、长期产品 |

**决策流程**：
```
数据是否敏感？
  是 → 评估本地部署成本是否可接受？
       是 → Build（私有化部署）
       否 → 数据脱敏后再 Buy
  否 → 当前阶段是否在验证期（MVP）？
       是 → Buy（快速试错）
       否 → 评估长期 TCO → 再决定 Build or Buy
```

## 七、上下文工程（Context Engineering）

> 从"提示词填充"演变为管理 Agent "认知边界"与"业务逻辑"的核心手段。

**核心观点**：好的上下文工程不是"给得越多越好"，而是在有限 Token 窗口内提供**最高密度的有效决策依据**。

### 7.1 四大核心维度

| 维度 | 内容 | 示例 |
|------|------|------|
| **静态上下文** | 角色设定、业务准则、安全护栏 | 企业合规政策、品牌话术规范 |
| **动态上下文** | 实时用户意图、任务进度、会话历史 | 客户购买记录、当前工单状态 |
| **知识上下文** | RAG 引入的私有知识 | 企业知识库、产品手册 |
| **过程上下文** | 中间思考过程、推理日志 | 多步决策的推理记录 |

### 7.2 常见 Agent 范式

```
ReAct（推理+行动）：先推理后行动，适合中等复杂度任务
计划与执行：高参数模型规划，低成本模型执行
状态感知检索：基于任务状态决定检索策略
层次化多 Agent：主管+员工结构，隔离上下文防污染
```

### 7.3 风险审计要点

```
- 技术债务：避免将复杂业务逻辑硬编码在 Prompt 中
- 成本核算：评估是否使用上下文压缩降低重复计算成本
- 安全性：在上下文流转中设置 PII 过滤，防止企业机密泄露
```

## 八、与工程师沟通技术可行性

**关键问题清单**：
```
1. 这个功能对模型推理的延迟（Latency）有什么要求？
2. 我们需要什么样的数据量才能启动冷启动？
3. 当前方案的并发上限是多少？
4. 如果用外部 API，数据是否会被用于模型训练？
5. 微调和 RAG 方案的实现周期各是多少？
6. 什么情况下模型会彻底答错？有没有兜底机制？
```

## 九、面试常见问题

**Q：AI产品的技术选型要考虑哪些因素？**
> 回答框架：7维度评估 → 架构方案选择（API/RAG/微调/Agent）→ 模型选型（公司属性/能力/成本/版权）→ Build vs Buy 决策

**Q：什么时候用 RAG，什么时候微调？**
> 回答框架：RAG适合知识更新频繁、需要溯源、数据不足的场景；微调适合有特定风格要求、大量标注数据、需要降低推理成本的场景

**Q：怎么判断一个场景是否适合用AI？**
> 回答框架：泛化性+数据量+容错率+成本。高泛化、数据足、可容错 → 适合；强确定、低容错、成本敏感 → 优先规则
