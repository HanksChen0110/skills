# 需规审校器

用于审校中文需求规格说明书，特别适合以下场景：

- 用户说“先检查一遍再改”
- 用户强调“严格按模板来”
- 用户反复指出 Word 排版、目录、表格、标题样式有问题
- 用户要求交付前做一次模板一致性和内容完整性检查
- 用户拿出两版 `.docx`，要求分析哪一版更接近正式交付稿，并把差异沉淀成长期规则

## 审校输入

- 正式模板 `.docx`
- 目标需求规格说明书 `.docx`
- 必要时补充来源材料，如 PRD、评审意见、调研稿、会议纪要

## 审校原则

- 先检查模板一致性，再检查章节完整性，再检查内容质量。
- 先找会影响交付的硬伤，再看措辞优化。
- 审校不是自由重写，除非用户明确要求改写内容。

## 核心检查项

### 1. 模板一致性

- 标题名称是否与模板一致。
- 标题层级、目录层级、章节顺序是否与模板一致。
- 表头列名是否与模板一致。
- 封面、目录、正文、表格的字体、字号、段前段后、行距是否与模板一致。
- 正文是否保持模板要求的首行缩进。
- 是否存在“视觉上像标题、实际是 Normal 正文”的伪标题。
- 是否存在“视觉上像题注、实际是 Normal 正文”的伪题注。

### 2. 目录污染

- 目录里只能有真实标题。
- 普通正文、说明句、图题、表题、列表项不得进入目录。
- 若目录里出现正文句子，通常说明该段落被错误赋予了标题样式或大纲级别。
- 若正文里出现 `4 合格性规定`、`5 需求可追溯性` 等章节，但目录没有这些项，通常说明文档在用普通正文伪装章节标题。

### 3. 悬浮小标题

- 检查是否存在无编号但承担分节作用的小标题。
- 典型问题是正文中突然出现 `核心功能模块边界`、`功能分解` 之类单独一行却没有编号。
- 这类内容若是正式小节，应升格为带编号标题。

### 4. 四级标题格式

- 四级标题如 `3.2.3.1 功能分解` 一律检查为 `黑体`。
- 不允许出现 `微软雅黑` 混入四级标题。
- 修正文档后要重新核查目录和标题层级。

### 5. 表格结构

- 表题必须在表格外部。
- 不允许把 `表 X ...` 写进表头或表格第一行。
- 不允许把前后两张表误连成一张表。
- 不允许出现“只有编号或标识、其他列为空”的孤立行。
- 检查有无串列、错位、误合并、空白首行、重复表头异常。

### 6. 功能描述质量

- 功能描述应说明场景、对象、动作、目的和结果。
- 审核时要判断其是否自然覆盖 5W + Effect，而不是只写成口号。
- 以下表述直接判为低质量：
  - “用于实现 xx 管理”
  - “负责 xx 相关业务处理”
  - “支撑 xx 业务开展”

### 7. 总分结构一致性

- 前文提出的系统、子系统、角色、边界、对象，后文是否一一承接。
- 后文是否出现前文没有定义的新名词、新模块、新边界。
- `3.2` 的子系统展开是否与 `3.1` 的总体框架一致。
- 前文是否先交代了业务总层，再交代系统切入点，再交代系统模块层；如果一上来只写系统自己，通常说明叙事总分关系不完整。
- 如果系统只赋能大业务链中的若干审核/确认节点，文档是否明确写出了“重点落点 + 延伸环节”，避免把系统写成覆盖整个业务系统。
- 若前文宣称三模块 / 四模块，后文 `3.2`、图题、表题、角色关系、动态关系是否全部同步。

### 8. 内容节奏与交付感

- 正式需规正文应以完整自然段为主，不应被大量草稿式单行句切碎。
- 同一章节中如果出现连续大量 `Normal` 短句承担标题、题注、说明、章节号多重角色，通常说明文档仍停留在“手工堆字”阶段，没有形成可交付的 Word 结构。
- 重点观察 `1.2` 与 `3.1`：优化后的交付稿通常表现为“段落更少但更密、层次更清楚、模块切入更自然”。

## 输出方式

默认按“问题清单”输出，按严重程度排序：

1. 破坏模板结构或版式一致性的问题
2. 缺失章节或错误层级的问题
3. 表格结构与表题问题
4. 目录污染和标题样式问题
5. 功能描述质量问题
6. 术语、命名、一致性问题

每条问题尽量指出：

- 位置
- 问题是什么
- 为什么不符合模板或需规写法
- 应如何修改

## 修订后复核清单

- 目录只包含真实标题
- 正文首行缩进未丢失
- 四级标题统一为 `黑体`
- 表题全部在表格外部
- 表格无孤立编号行、无误合并、无错位
- 前后文命名一致
- 章节完整，可直接进入交付或下一轮定稿
- 业务总层与系统模块层已经分开表达
- `1.2` / `3.1` 已形成“业务全貌 -> 切入节点 -> 典型场景 -> 底座与差异化”的顺序或等价顺序
- 文档没有再用普通正文伪装章节、图题、表题
