# 第 3 章模板规则

## 适用范围

用于生成或续写需求规格说明书第 3 章，尤其是 `3.2 CSCI能力需求`。

## 一、格式继承规则

当存在原始需规、`docx` 或既有章节时，默认按原文格式续写，至少继承以下要素：
- 章节编号层级
- 标题命名风格和标识命名
- 图题、表题、表号和题注样式
- 表号是否通过题注或域自动生成
- 表格表头、列顺序、对齐方式和宽度关系
- 正文样式、首行缩进、行距和段前段后
- 角色名称、系统名称、子系统名称和业务对象名称

禁止事项：
- 擅自改变 `3.2` 的组织方式
- 擅自重排性能需求位置
- 擅自改变图表编号机制
- 用 Markdown 风格替代 Word 正文样式
- 在同一章节中混用手工编号和自动编号

## 二、`3.2` 子系统展开规则

### 1. 同级条目统一为子系统

从 `3.2.1` 开始的同级条目，默认都按并列子系统组织：
- `3.2.1 {子系统A}`
- `3.2.2 {子系统B}`
- `3.2.3 {子系统C}`

写法要求：
- 不区分“本体类结构”“智能体类结构”“孪生体类结构”三套层级。
- 若确有本体、智能体等说法，仅作为子系统名称出现，不改变层级规则。
- 多个智能能力可合并写成一个“智能辅助子系统”，在其下再拆功能和用例。

### 2. 性能需求位置

- 性能需求不是固定的 `3.2.2`。
- 只有在全部同级子系统写完后，或原文明确如此安排时，才编写性能需求。
- 若原文已有性能需求位置，严格继承原文。

## 三、功能分解规则

### 0. 功能描述统一写法

文档中所有层级的功能描述均遵循同一标准，包括一级模块概述、二级模块描述、三级子功能描述，以及功能划分一览表、功能分解表中的描述性文字。

统一要求如下：
- 使用一整段连贯文字，不分条列举。
- 正文中不出现“服务对象”“业务场景”“核心能力”“价值目的”“异常机制”等维度名称。
- 内容应自然客观地覆盖六个方面：
  1. 该模块服务于哪个角色；
  2. 在什么业务环节、什么时机被触发使用；
  3. 用户通过什么方式完成什么操作；
  4. 系统在后台如何处理、具备哪些能力；
  5. 解决了什么业务痛点、达到了什么目的；
  6. 出现异常时系统如何响应。

推荐写法：
本模块主要服务于[角色名]，旨在解决[痛点描述]的业务痛点。在[业务环节]，用户通过[操作入口]录入或处理[核心信息]。系统应具备[核心能力1]和[核心能力2]能力，支持用户完成[关键操作]，并基于[规则/逻辑]自动处理，实现[预期效果]。当出现[异常情况]时，系统应能够[异常响应措施]。

子系统通常按以下顺序展开：
1. 子系统概述
2. 功能分解
3. 用例分解

“功能分解”表在无更强原文约束时通常包含：
- 序号
- 子功能名称/标识
- 功能描述
- 优先级

功能描述要求：
- 单条通常控制在 `200-300` 字
- 按“角色、场景、操作、系统处理、业务价值、异常响应”六方面自然展开
- 不直接把这些维度名写入正文
- 不写成并列短句堆砌，优先写成一整段连贯说明

## 四、用例分解规则

### 1. 管理类功能必须覆盖增删改查

以下类型默认视为管理类功能：
- 台账
- 目录
- 分类
- 关系
- 状态
- 配置
- 规则
- 模板
- 流程维护
- 经验维护

最少条目要求：
- 新增
- 删除/作废
- 修改
- 查询

可按业务增加：
- 审核
- 导入
- 导出
- 启用
- 停用
- 提交
- 回退
- 同步

注意：
- 扩展条目不能替代增删改查
- 若无物理删除，应用“作废/停用/失效”等等价条目并显式说明

### 2. 用例目标写法

- 建议按操作步骤分条展开
- 每条单独一行
- 优先用“表单录入：”“数据校验：”“状态初始化：”“审批提交：”等动词引导

### 3. 异常处理写法

异常处理优先写为：

`当出现 XX 情况时，系统应能够 XX。`

## 五、性能需求规则

若原文已有性能需求表或指标条目：
- 默认保持原文内容不变
- 保持表头、表题、位置和指标口径不变
- 不从其他项目迁移性能指标

只有以下情况才允许修改：
- 用户明确要求重写性能需求
- 用户明确要求替换性能指标
- 原文性能需求为空，且用户明确要求补全

## 六、编号与命名规则

- 子系统名称与标识必须成对出现
- 同一子系统内的子功能标识应连续且风格统一
- 用例标识应与对应子功能标识关联

若原文无统一编号体系，默认采用：

`GN-[系统缩写]-[子系统缩写]-[子功能缩写]-[用例缩写]`

## 七、表格与题注规则

若当前章节使用“功能一览表”“用例一览表”等表格：
- 表题在无更强原文约束时，统一使用 `表X.X.X 表格名称`
- 表题单独成行并居中
- 若原文使用 Word 题注或域编号，续写时必须继续沿用

当条目属于智能辅助、外部接口、硬件集成或后续扩展能力时：
- 本期完成交付：正常描述
- 架构预留：备注写“本期完成架构预留，后续版本补全交付”
- 不纳入本期：备注写“本期不纳入建设范围，预留接口，后续版本扩展”

## 八、`3.2.2` 及以后版式保护规则

续写现有 `docx` 时，`3.2.2` 及以后内容必须额外检查：
- 同级标题是否沿用原文同一标题样式
- 新增正文是否沿用原文正文段落样式
- 题注是否沿用原文题注样式和编号机制
- 表头结构、列顺序、对齐方式是否与同类原表一致
- 是否出现“手工新造表题”“标题能对但样式不对”的情况

若无法稳定继承原文版式，应先说明风险，不得假装已保真。
