# 第 3 章标准骨架

以下骨架用于无明确原文时的默认生成；若用户提供了现有需规，以现有文档结构为准。

```markdown
3 需求/GN
3.1 要求的状态和方式
3.1.1 要求的状态和方式说明/YF
3.1.2 功能框架及关系描述
3.1.3 状态和方式下能力配置/NP
3.2 CSCI能力需求
3.2.1 {子系统A名称}/{子系统A标识}
3.2.1.1 功能分解
3.2.1.2 用例分解
3.2.2 {子系统B名称}/{子系统B标识}
3.2.2.1 功能分解
3.2.2.2 用例分解
3.2.x {其他并列子系统}
3.2.n 性能需求
3.2.n.1 性能指标/XN
```

## 默认组织方式

- 从 `3.2.1` 开始的同级条目，默认全部视为并列子系统。
- 若存在多个智能能力，默认可合并为一个“智能辅助子系统”后再展开。
- 不默认把“本体”“智能体”“孪生体”写成不同结构层级。
- 不默认把 `3.2.2` 写成“性能需求”。

## 管理类子功能最少用例

对于典型管理类子功能，至少写出以下 4 条：

| 序号 | 角色 | 用例（事件）名称/标识 | 用例目标 |
| --- | --- | --- | --- |
| 1 | 管理角色 | 新增XX/{子功能标识}-01 | 建立新对象记录。 |
| 2 | 管理角色 | 删除或作废XX/{子功能标识}-02 | 删除或作废对象记录。 |
| 3 | 管理角色 | 修改XX/{子功能标识}-03 | 更新对象信息。 |
| 4 | 使用角色、管理角色 | 查询XX/{子功能标识}-04 | 查询和定位对象记录。 |

可按业务继续增加审核、导入、导出、启停、提交、回退等条目。

## 性能需求默认处理

- 若原文已有性能需求表，保持原表不动。
- 若原文无性能需求且用户明确要求补全，再在所有同级子系统之后单独编写性能指标表。
