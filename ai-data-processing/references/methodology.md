## 一、数据处理在 AI 产品中的地位

数据是 AI 产品的核心燃料。**模型的上限由数据质量决定**，而非模型本身。

**谁来负责数据处理？**
| 公司规模 | 主要负责人 |
|---------|-----------|
| 上规模公司 / 专业模型厂 | 数据标注师 / 数据运营团队 |
| 常规公司 | **产品经理** 承接（这是 AI PM 的核心差异化能力） |

## 三、数据来源识别

### 3.1 常见数据来源

| 来源类型 | 示例 | 特点 |
|---------|------|------|
| **历史业务数据** | 客服问答记录、工单系统 | 最贴近真实场景，质量参差不齐 |
| **企业文档** | PDF、Word、飞书文档、知识库 | 结构化程度不一，需解析处理 |
| **数据库/API** | ERP、CRM 系统数据 | 结构化，需定义接入方式 |
| **网络数据** | 公开文章、行业报告 | 需注意版权，质量需筛选 |
| **合成数据** | 用大模型批量生成 | 数据不足时的补充方案 |

### 3.2 数据充足性评估

```
数据是否足够？需回答：
1. 有多少条历史数据？（数量）
2. 数据覆盖了哪些场景/类型？（覆盖度）
3. 数据的标注质量如何？（质量）
4. 数据的时效性如何？（时效）
```

**数据不足时的应对方案：**
```
方案A：用大模型批量生成合成数据
  → 先通过用户调研确认问题分类和语气特征
  → 再用 LLM 基于模板批量生成 Q&A 对

方案B：冷启动阶段用规则+人工先运行
  → 积累真实用户数据后再迭代
```

## 四、数据加载（Data Loader）

### 4.1 PDF 文档处理（重点）

PDF 是最常见也最复杂的文档格式，分三种类型：

| PDF 类型 | 特点 | 处理方式 |
|---------|------|---------|
| **机器生成 PDF** | 文本可选中、可搜索，由 Word/Typora 等导出 | 直接解析文本层 |
| **带 OCR 的扫描文档** | 扫描后用 OCR 识别，可能有识别误差 | 解析 + 校验 OCR 质量 |
| **传统扫描文档** | 纯图片 PDF，文字无法选中 | 需先跑 OCR，再提取文本 |

> ⚠️ **注意**：OCR 仍有一定识别错误率，不能完全信任，需要人工抽查质量。

### 4.2 其他格式

```
- Word / Markdown：相对简单，直接解析
- 数据库：通过 SQL 接口定义查询逻辑
- API 数据：定义接口调用规则，处理分页和频率限制
- 图片：需要多模态模型（Vision）或 OCR 处理
```

## 五、数据清洗（Data Cleaning）

**目标**：提高数据的准确性和一致性，减少噪声。

### 5.1 去除重复数据
```
方法：使用哈希值或唯一标识符检测并删除重复记录
tool：Python pandas dedup、数据库唯一约束
```

### 5.2 纠正错误
```
- 修复拼写错误、格式不一致
- 统一术语表达（如"AI"和"人工智能"代表同一概念）
- 纠正标注错误（人工核查抽样）
```

### 5.3 处理缺失值

| 处理方式 | 适用场景 |
|---------|---------|
| **删除** | 缺失比例低，且该条记录完整性低 |
| **填充均值/中位数** | 数值型字段 |
| **插值法** | 时序数据（拉格朗日插值、牛顿插值） |
| **预测填充** | 用模型预测缺失值（成本高，重要字段才用） |

### 5.4 异常值检测与处理
```
统计方法：箱线图（IQR）、Z-score
机器学习：孤立森林（Isolation Forest）
处理方式：删除 / 替换 / 调查确认原因
```

## 六、数据变换（Data Transformation）

**目标**：将数据转换为适合 AI 处理的标准格式。

### 6.1 格式统一

```
日期格式  → 统一为 ISO 8601（YYYY-MM-DD）
货币格式  → 统一为小数点表示（1000.50）
单位格式  → 统一为国际单位制（kg、m、s）
```

### 6.2 文本标准化

```
- 去除特殊字符（HTML 标签、控制字符）
- 去除多余空格、换行符
- 统一大小写（英文场景）
- 繁简体转换（中文场景）
- 去除无意义停用词（视场景决定，RAG 场景慎删）
```

### 6.3 分类编码

```
标签编码（Label Encoding）：文本标签 → 唯一整数
  示例："正面" → 1，"负面" → 0

独热编码（One-Hot Encoding）：文本标签 → 二进制向量
  示例："产品咨询" → [1,0,0]，"售后问题" → [0,1,0]
```

## 七、元数据提取

**非常关键！** 元数据是 RAG 检索精度的重要提升手段。

```
需提取的元数据：
- 文件名、文件类型
- 创建时间、修改时间
- 章节标题（H1/H2/H3）
- 图片 alt 属性
- 来源 URL / 文档 ID
- 作者信息
- 关键词标签
```

> **为什么重要**：元数据可用于过滤检索范围（如"只检索 2025 年后的文档"），大幅提升召回精度。

## 八、分块策略（Chunking）

分块策略直接影响 RAG 的召回质量。

### 8.1 常见分块方式

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **固定大小分块** | 按字数固定切割（如每块 500 字） | 快速实现，效果一般 |
| **语义分块** | 按段落/章节边界切割 | 保留语义完整性，推荐 |
| **句子滑窗** | 相邻块有重叠（防止上下文截断） | 信息连续性强的场景 |
| **父子分块** | 小块用于精确检索，大块用于上下文 | 平衡精度与完整性 |

### 8.2 分块大小建议

```
- 太小（< 100 字）：语义不完整，上下文丢失
- 太大（> 1000 字）：引入噪声，降低相关性
- 推荐范围：300-800 字，根据具体场景调整
- 建议设置 10-20% 的重叠（overlap）避免截断
```

### 8.3 面试常见问题

> Q：你们的 chunk 策略是什么？为什么这样设计？
> A：根据文档结构按语义段落切割，大小控制在 500 字左右，相邻块设置 10% 重叠。这样既保证语义完整性，又避免单块内容过长引入噪声。

## 九、向量化（Embedding）

### 9.1 流程

```
输入文本 → 分词 → Embedding 模型处理 → 生成向量 → 存入向量数据库
```

### 9.2 Embedding 模型选择

```
推荐参考：
- HuggingFace MTEB 排行榜（中文/英文分开看）
- 推荐：BGE 系列（BAAI/bge-large-zh-v1.5）在中文场景表现优秀
- 也可参考 QAnything 的 embedding 方案
```

### 9.3 检索方式

| 方式 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **向量检索（语义）** | 计算向量相似度 | 能处理语义相近但词汇不同的情况 | 关键词精确匹配弱 |
| **关键词检索（全文）** | BM25 等算法 | 精确匹配关键词 | 无法理解语义 |
| **混合检索** | 两者结合 | 兼顾精度和语义 | 需调权重参数 |

### 9.4 检索调优指南

```
问题 → 解决方案

用户问题与文档有关键词联系    → 提高关键词相似度权重 / 用全文检索
用户问题与文档仅语义相近      → 用向量检索 / 降低关键词权重
文档中有大量相似文本          → 提高混合相似度阈值
非英语外语知识库              → 默认使用向量检索（且不可更改）
```

## 十、RAG 召回率优化

### 召回率计算

```
召回率 = 检索到的相关文档数 / 知识库中相关文档总数

F1 综合指标：
  Precision（精确率）= 检索结果中相关文档 / 检索结果总数
  Recall（召回率）  = 检索到的相关文档 / 实际相关文档总数
  F1 = 2 × (P × R) / (P + R)
```

### 优化方向

```
1. 改进检索模型（换更好的 Embedding 模型）
2. 优化知识库结构（分块策略、元数据）
3. 使用 ReRank 重排（粗排→精排二阶段）
4. Query 改写（联系上下文理解用户真实意图）
5. 调整 TopK 参数（召回数量）
```

## 十一、数据标注

### 标注工作内容

```
- 意图分类标注：给用户问题打标签（产品咨询/投诉/建议等）
- 质量评分标注：对 AI 回答进行好/中/差打分
- 实体标注：标记文本中的关键实体（人名/地名/产品名）
- 对话对标注：整理 (question, answer) 训练对
```

### 标注质量控制

```
- 双人标注 + 一致性检验（Kappa 系数 > 0.7 才可用）
- 建立标注规范手册，减少主观分歧
- 定期抽查，发现系统性偏差及时纠正
```
